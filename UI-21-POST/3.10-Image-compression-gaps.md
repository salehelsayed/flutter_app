  Where it's broken — preference silently ignored

  All these gaps default to compressed (quality 85) regardless of the user's actual
  setting:

  Gap 1: Notification tap → 1:1 conversation (main.dart:~1350)
  ConversationWired is constructed without imageProcessor or qualityPreference. Since
  imageProcessor is null, _processMediaIfNeeded() short-circuits and returns the raw
  unprocessed file — no EXIF stripping, no compression, nothing.

  Gap 2: Notification tap → group conversation (main.dart:~1322)
  GroupConversationWired is constructed without qualityPreference. Falls back to the
  constructor default of compressed.

  Gap 3: Posts surface → "message author" conversation (posts_wired.dart:~967)
  ConversationWired is opened with imageProcessor but without qualityPreference. Always
  defaults to compressed even if the user chose original.

  Gap 4: OS share sheet → all conversations (share_target_picker_route.dart)
  The buildShareTargetPickerRoute function doesn't even accept a quality preference
  parameter. Every ShareTargetPickerWired instance — and every
  ConversationWired/GroupConversationWired it pushes — is stuck at compressed.

  Affected entry points:
  - main.dart:~1197
  - qr_scanner_wired.dart:~462
  - first_time_experience_wired.dart:~558
  - startup_router.dart:~669

  Minor inconsistency

  orbit_wired.dart:~309 — _loadQualityPreference() assigns without setState(), unlike
  feed_wired.dart which correctly uses setState. This works in practice because the value
  is only read at navigation-push time, but it's inconsistent.

  Summary

  The core processing pipeline (ImageProcessor.processImage + loadImageQualityPreference)
  is solid. The problem is purely in DI threading — several navigation entry points
  construct conversation screens without passing the loaded preference. The fix for each
  gap is the same pattern: load the preference from SecureKeyStore and pass it through the
  constructor.
