# Phase 5: Original-Author Avatar Snapshot — TDD Orchestration Plan

Prepared on: 2026-03-17
Depends on: `UI-21-POST/3.6-Repost-media-improv.md` Phase 5

## Objective

Make reposted posts self-renderable for original-author identity by carrying
a compact avatar snapshot inside the encrypted repost payload, persisting it
in Posts-owned durable state, and teaching PostCard to prefer that snapshot
over the file-on-disk/RingAvatar fallback.

## Current State Summary

| Layer | Current Behavior | Gap |
|-------|-----------------|-----|
| `RenderablePostSnapshot` | Carries `authorPeerId` + `authorUsername`, NO avatar data | No avatar bytes in repost payload |
| `PostPassEnvelope._buildPayloadJson()` | Serializes snapshot without avatar field | Avatar not transmitted |
| `PostModel` | Has `authorPeerId`, no avatar bytes field | Cannot carry avatar from DB to UI |
| `post_origin` / DB | Stores `passer_peer_id`, `passer_username` — no avatar column | No durable avatar snapshot persistence |
| `posts_db_helpers` feed queries | LEFT JOIN `post_origin` for passer — no avatar join | Feed load cannot hydrate avatar bytes |
| `UserAvatar` widget | Priority: `avatarBytes` > file on disk > RingAvatar > fallback | Already supports `avatarBytes` param — just needs caller to supply it |
| `PostCard` | `UserAvatar(peerId: post.authorPeerId)` — no avatarBytes passed | Falls back to RingAvatar for non-contact authors |
| `passPostAlong()` | Loads post + media, builds envelope — no avatar loading | Sender doesn't read original author's avatar |
| `handleIncomingPassedPost()` | Creates PostModel from envelope — no avatar storage | Receiver doesn't persist avatar snapshot |
| `post_surface_hydrator` | Loads media, comments, reactions — no avatar hydration | Avatar bytes not part of surface model |
| Existing tests | `post_pass_envelope_test.dart` line ~115 explicitly asserts NO avatar keys | Tests lock the broken baseline |

## Locked Design Decisions

1. **Avatar field name**: `original_author_avatar_base64` in JSON, `originalAuthorAvatarBase64` in `RenderablePostSnapshot`, `originalAuthorAvatarBytes` (Uint8List?) in `PostModel`
2. **Avatar location in payload**: Inside `original_snapshot` object (not top-level payload), because the avatar is an attribute of the original author's identity snapshot
3. **Avatar encoding**: base64-encoded JPEG bytes in the wire envelope (compact, bounded)
4. **Avatar size bound**: Max 64 KB base64 (approximately 48 KB raw JPEG — larger than the 512x512 q80 avatars the app produces). Sender silently omits avatar if bytes exceed the bound rather than failing the repost
5. **DB persistence**: New table `post_pass_avatar_snapshots(post_id TEXT PRIMARY KEY, author_peer_id TEXT NOT NULL, avatar_blob BLOB, created_at TEXT NOT NULL)` — BLOB storage avoids base64 decode/encode overhead on every feed load
6. **Feed hydration**: `post_surface_hydrator` loads avatar bytes in bulk for the current feed page, attaches them to `PostModel.originalAuthorAvatarBytes` via `copyWith`
7. **PostCard change**: When `post.passedByPeerId != null && post.originalAuthorAvatarBytes != null`, pass `avatarBytes: post.originalAuthorAvatarBytes` to `UserAvatar`
8. **Fallback**: When avatar snapshot is absent (legacy reposts, oversized avatar), `UserAvatar` falls back to file-on-disk → RingAvatar as today — no behavior change
9. **Idempotency**: Duplicate `passId` delivery does not overwrite or corrupt stored avatar (INSERT OR IGNORE / upsert with same content)
10. **No contact mutation**: Persisting the avatar snapshot does NOT create or modify the receiver's contact list. The snapshot lives in Posts-owned state only

```dart
final Map<String, Uint8List> _passAvatarSnapshots = {};

@override
Future<void> savePassAvatarSnapshot(String postId, String authorPeerId, Uint8List avatarBlob, String createdAt) async {
  _passAvatarSnapshots.putIfAbsent(postId, () => avatarBlob);
}

@override
Future<Uint8List?> loadPassAvatarSnapshot(String postId) async {
  return _passAvatarSnapshots[postId];
}

@override
Future<Map<String, Uint8List>> loadPassAvatarSnapshotsForPosts(List<String> postIds) async {
  return {
    for (final id in postIds)
      if (_passAvatarSnapshots.containsKey(id))
        id: _passAvatarSnapshots[id]!,
  };
}
```

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Avatar BLOB in DB bloats feed load | Hydrator loads avatars separately in bulk (not in feed SQL query). BLOBs are max 48 KB raw. |
| Large avatar increases encrypted payload size | 64 KB hard bound. Sender silently omits oversized avatar. Receiver renders RingAvatar fallback. |
| Agent 4 and Agent 5 have shared files (PostModel, PostRepository) | Agent 4 handles model field addition + repository interface. Agent 5 handles envelope + use case + UI. Clear boundary: Agent 4 owns the persistence seam, Agent 5 owns the behavior seam. |
| Existing test assertion `snapshot.keys.where((key) => key.contains('avatar')), isEmpty` will fail after avatar is added | Agent 1 must update this assertion to expect the avatar key when present, and keep it absent-only for the no-avatar case |
| Non-contact original author renders with stale avatar after real avatar changes | Acceptable: the snapshot is point-in-time. The repost card shows the avatar at repost time, not a live-updating avatar. Consistent with how username is snapshot-frozen. |
