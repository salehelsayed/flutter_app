## Goal

Make repost visually legible on the card itself.

When User-A reposts a post of User-B to one or more recipients:

- the repost action on User-A's card should switch into an active visual state
- User-A should see a numeric label next to that repost action
- User-B should also see a numeric label on the original card
- any receiver who gets the passed-along card with a carried repost-total
  baseline should see that aggregate count on the reposted card

The desired outcome is:

- the repost action is no longer visually static after a successful local repost
- the number shown reflects people selected for repost, not just count of pass
  events
- the reposter card and the original-author card both update from durable
  Posts-owned state
- a passed-along receiver card can also render the existing repost total that
  traveled with the repost, and keep updating from later known repost events

## Product Assumption Locked In This Plan

This plan makes one explicit product assumption so implementation stays bounded:

- on the reposter's own card, show the viewer-local number of people they
  shared that post to across their local repost events
- on the original author's card, show the aggregate number of people the post
  has been shared to across all repost events known to that author
- on a passed-along receiver's card, show the aggregate number of people the
  post has been shared to when that total is available from carried baseline
  plus later known repost events, but keep the control visually neutral unless
  the viewer is also the author or reposter

This plan intentionally does **not** attempt a privacy-invasive unique-recipient
ledger. The count is defined as the sum of explicit repost-recipient selections
per accepted repost event, excluding the original-author notification delivery.

That means:

- if a repost event targets three selected recipients, it contributes `3`
- if the original author also receives a notification copy, that notification
  does not add `+1`
- if the same post is reposted again later, the later repost adds its own
  recipient count

This preserves privacy better than carrying full recipient identity lists just
to compute a cross-pass unique-person count.

## Current Findings

### UI rules do not support this yet

- `PostCard` already has a repeat action, but its color is currently static
- `PostsScreen` only shows the share metric for the original author
- receivers who are not authors do not see any repost count, even when the
  repost already carries a meaningful historical total
- there is no viewer-local repost-active flag exposed to the widget layer

### The current share metric is the wrong metric

- `PostModel.shareCount` is currently derived from `COUNT(*)` over `post_passes`
- that means it counts repost events, not people shared to
- the requested product metric cannot be expressed accurately with the current
  field

### Persistence cannot represent the new metric yet

- `post_passes` currently stores no `recipient_count`
- the repost send path already knows the resolved recipient set before delivery
  starts, but that number is discarded instead of being persisted
- incoming repost notifications also do not currently carry an explicit
  `recipient_count`
- there is no durable carried repost-total baseline for a fresh repost
  recipient's copy, so even a correct incoming repost cannot currently project
  the original total on the receiver card

### Viewer-local visual state cannot be derived cleanly yet

- `PostModel` has no `viewerHasPassed`-style field
- the card therefore cannot reliably distinguish:
  - the original author seeing aggregate repost activity
  - the reposter seeing their own reposted state
  - a passive receiver simply viewing a passed-along post

## Locked Decisions

- coordinate with `3.6` for carried repost-total baseline transport and
  persistence, but keep the metric projection and UI rules in this `3.7` plan
- do not overload `shareCount` with a second meaning
- introduce explicit repost-visual metrics in the Posts surface model:
  - `totalSharedToCount`
  - `viewerSharedToCount`
  - `viewerHasPassed`
- define `totalSharedToCount` as the carried repost-total baseline from repost
  snapshot state plus later repost-event deltas known locally for that post, or
  the smallest equivalent authoritative projection
- define `viewerSharedToCount` as the sum of `recipient_count` across the
  current viewer's local outgoing repost events for that post
- define `viewerHasPassed` as `viewerSharedToCount > 0`
- add `recipient_count` to the repost persistence contract
- exclude original-author notification from `recipient_count`
- when a passed-along receiver has a carried or reconstructed aggregate repost
  total for the post, show that count on the card in a neutral visual state
- make the repost action visually active on:
  - the reposter's card when `viewerHasPassed == true`
  - the original author's card when `totalSharedToCount > 0`
- keep passive receiver cards visually neutral even when they show the carried
  aggregate repost count
- keep the visual change scoped to the repost action itself; do not redesign the
  whole card
- preserve existing repost durability, one-hop limit, `pickPeople` rejection,
  and retry semantics
- keep repost counts driven from durable Posts state, not transient in-widget
  state
- keep author-notification privacy bounded:
  - carry only the numeric `recipient_count`
  - do not add full recipient identity lists just to render the metric

## Non-Goals

This plan does not cover:

- repost sender-speed changes from `3.5`
- repost media/avatar/encryption work from `3.6`
- redesigning the full repost card
- changing who can repost
- changing follow-on or repost fanout concurrency
- computing a globally unique historical recipient count across all reposts of a
  post
- changing the heart metric contract; heart-baseline transport and stateful
  continuity belong to `3.6`

  that author is excluded from `recipient_count`

- incoming repost handling persists `recipient_count` from the incoming repost

  - viewer-local `viewerSharedToCount`
  - `viewerHasPassed`
- `totalSharedToCount` on the original author's copy aggregates reposts from
  multiple reposters and multi-recipient repost events by recipient total, not
  by repost-event count
- `totalSharedToCount` for a passed-along receiver combines the carried
  baseline plus later locally known repost deltas instead of restarting from
  zero/one
- legacy repost rows with no `recipient_count` remain readable via a compatible
  fallback value

- extend repost send and receive contracts to carry `recipient_count`
- persist the resolved explicit-recipient count at local repost creation time
- persist or project the carried repost-total baseline delivered by `3.6`
- add repository/query support for:
  - aggregate shared-to count across repost events
  - aggregate shared-to count across multiple reposters on the original

### Exit Gate

Outcome:
- the new visual state updates promptly and does not regress repost correctness

Suggested test files:

Minimum RED cases:
- after a local repost is persisted, the reposter's card refreshes to the new
  active visual state without waiting for full network settlement
- when the original author later receives the repost notification, their card
  refreshes to the new aggregate count
- when two different direct friends repost the same post to different recipient
  counts, the original author's card refreshes to the summed aggregate total
- when a passed-along receiver loads a repost with a carried aggregate baseline,
  their card refreshes to that total without waiting for a later repost event
- when later repost events arrive, the aggregate count shown on eligible cards
  continues from the carried baseline rather than snapping back to locally
  created rows only
- retrying or reprocessing the same repost event does not increment the metric
  twice
- compatibility rows created before `recipient_count` existed still render a
  stable fallback metric instead of disappearing
- the visual-state work does not break passed-along banners, existing share
  attribution, or feed ordering

### GREEN

- wire the new metric fields into the existing repository-driven post refresh
  path
- preserve repost idempotency and compatibility behavior while layering on the
  new UI state
- preserve neutral-vs-active styling rules while allowing carried totals to
  surface on passed-along receiver cards

### REFACTOR

- keep refresh behavior data-driven through `postChanges` and repository reloads
  rather than widget-local bookkeeping

### Exit Gate

- visual updates are prompt, durable, and retry-safe

## Residual Risks To Track During Implementation

- historical repost rows cannot recover the exact original recipient-set size if
  they were created before `recipient_count` existed, so compatibility fallback
  will be approximate
- "people shared to" in this plan means summed recipient counts per repost
  event, not globally deduped unique people across all repost history
- passive receiver cards now depend on carried repost-total baseline from `3.6`;
  if that baseline is missing for legacy rows, compatibility fallback will be
  necessarily approximate until a newer repost arrives
