## Why This Needs A New Plan

The repost avatar bug is now narrowed enough that it should live in its own
phase doc instead of being folded back into `3.6` or `3.8`.

- `3.6-Repost-media-improv.md` established repost media and avatar transport
- `3.8-Repost-author-fidelity-and-thread-continuity.md` proved the exact
  `A↔B`, `B↔C`, no `A↔C` topology and isolated the avatar failure to the
  sender-side omit branch
- the remaining gap is more specific:
  - the app already has one avatar-processing contract via
    `ImageProcessor.processAvatar(...)`
  - repost sender does not currently use that contract
  - contact-avatar download also does not currently use that contract

That means the app can store a valid local avatar that renders fine in normal
surfaces, but still omit that same avatar from a repost because the repost path
checks raw file bytes instead of canonical processed avatar output.

Suggested file:
- `UI-21-POST/3.9-Repost-avatar-processAvatar-alignment.md`

## Goal

Make repost avatar transport follow the same client-side avatar contract as the
rest of the app.

When user B reposts user A's post to user C:

- B should prepare A's avatar through `processAvatar()`
- that client-side processing contract should remain:
  - `512x512`
  - quality `80`
  - EXIF stripped
- C should receive and render A's real avatar on the repost card
- the repost path should no longer drop a valid local avatar merely because the
  pre-processed file on disk is larger than the old raw-byte cap

The desired outcome is:

- client-owned avatar files are normalized through one canonical processing
  contract
- repost avatar snapshots are built from processed avatar output, not raw disk
  bytes
- previously downloaded or already-cached oversized avatars do not keep
  breaking repost fidelity
- device logs identify whether avatar preparation failed because of missing
  file, processing error, unreadable bytes, or some final transport safeguard

## Exact User-Reported Regression

- user A is friends with user B
- user B is friends with user C
- user A is not friends with user C
- A creates a post
- B receives it
- B reposts it to C
- C receives the post but not A's profile picture
- expected: C sees A's real profile picture on the repost card

## Current Findings

### The latest device log proves sender-side omission, not missing-file failure

On user B's device, repost creation logs:

- `POST_PASS_AVATAR_LOAD_START` with `loaderPresent: true`
- `POST_PASS_AVATAR_OMITTED_TOO_LARGE`
- `avatarByteLength: 121369`
- `maxBytes: 65536`

That means B did find and load A's avatar bytes from local disk, but then
intentionally omitted them from the repost payload because they exceeded the
current raw-byte bound.

### Upload already has a canonical client-side avatar processing contract

The current profile-picture pick/upload flow already uses:

- `settings_wired.dart` to call `imageProcessor.processAvatar(...)`
- `ImageProcessor.processAvatar(...)` to enforce:
  - `512x512`
  - quality `80`
  - `keepExif: false`
- `upload_profile_picture_use_case.dart` to copy the processed file into
  `media/avatars/<peerId>.jpg`

So the app already has an explicit avatar-processing contract. Repost is simply
not reusing it.

### Contact-avatar download stores raw relay bytes today

`download_profile_picture_use_case.dart` currently downloads avatar bytes from
the relay directly into `media/avatars/<peerId>.jpg` and updates the contact.
It does not run `processAvatar()` first.

That means:

- newly uploaded self avatars are locally processed on the originating device
- remotely downloaded contact avatars are not guaranteed to be locally
  normalized on the receiving device
- repost sender can therefore encounter oversized or EXIF-bearing contact
  avatar files in normal app usage

### Repost sender uses raw local bytes and a separate hard bound

`posts_wired.dart` currently loads avatar bytes from:

- `media/avatars/<peerId>.jpg`

`pass_post_along_use_case.dart` then:

- includes the avatar only when raw bytes are `<= 65536`
- emits `POST_PASS_AVATAR_OMITTED_TOO_LARGE` otherwise

That means the repost transport contract is currently:

- raw local file bytes
- plus a hard omit bound

instead of:

- processed avatar output from `processAvatar()`

### The current bug can persist even after fixing future downloads

Even if contact-avatar download starts processing avatars going forward, devices
may already have previously downloaded oversized avatar files on disk.

So this plan must cover both:

- future local avatar normalization on download
- repost-time protection for existing local avatar files that predate the fix

## Locked Decisions

- `ImageProcessor.processAvatar(...)` is the canonical client-side avatar
  processing contract for repost avatar snapshots
- that contract remains:
  - `512x512`
  - quality `80`
  - EXIF stripped
- do not use raw local avatar bytes as the repost transport contract
- do not use the pre-processed raw file size as the primary inclusion gate
- newly downloaded contact avatars should also be normalized client-side through
  the same processing contract before becoming the canonical local avatar file
- repost sender must still protect legacy/local stale avatar files by preparing
  a processed avatar snapshot at send time or through a shared normalization
  helper
- keep `original_author_avatar_base64` as the repost payload field unless tests
  prove an envelope change is necessary
- keep avatar persistence and hydrate behavior inside Posts-owned state
- add explicit FLOW diagnostics for:
  - local avatar path resolution
  - avatar processing start
  - avatar processing success
  - avatar processing failure
  - avatar snapshot include/omit result
- preserve one-hop repost depth, retry semantics, and current repost media
  behavior

## Non-Goals

This plan does not cover:

- repost-thread trust/fanout fixes from `3.8`
- repost visual-state work from `3.7`
- generic avatar redesign or `UserAvatar` styling changes
- changing relay avatar schema
- a broad media-pipeline rewrite outside avatar normalization
- new friends-of-friends contact behavior

## Residual Risks To Track

- repeated reprocessing of already-normalized JPEG avatars may slightly change
  bytes even if visual output remains correct
- platform-specific image-compression output can vary, so tests should assert
  behavior and contract rather than exact byte-for-byte image identity
- temp-file cleanup must remain safe on repeated repost retries
- the final processed avatar output may still need an explicit transport
  safeguard for pathological inputs; if so, that rule must be applied only
  after `processAvatar()` and be visible in logs and tests
