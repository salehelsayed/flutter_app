## Goal

Fix the biggest receiver-side comment UX gap and reduce delivery tail latency:

- an open comments sheet on the receiver must live-update when remote comments
  arrive, without requiring the sheet to be closed and reopened
- post-create bounded fanout concurrency must increase from `4` to `25`
- comment follow-on bounded fanout concurrency must increase from `4` to `25`

This plan is narrowly scoped to comment freshness and the requested concurrency
changes. It does not redesign the Comments UI and it does not redefine sender
settlement semantics.

## Current Findings

### Receiver comment freshness

- incoming comments are persisted immediately through the normal
  router -> listener -> use-case path
- comment persistence emits a `postChanges` event, so the Posts surface can
  reload promptly
- the feed card comment count can refresh quickly because `PostsWired`
  subscribes to `postRepo.postChanges`
- the open `CommentsSheet` is snapshot-only:
  - `PostsWired._openComments()` loads comments once, then passes a fixed list
    into the modal
  - `CommentsSheet` copies that list into local widget state
  - the sheet only refreshes after its own local submit/heart callbacks
- no UI currently consumes `PostCommentListener.incomingCommentStream`

### Sender-side bounded concurrency

- post-create fanout currently uses `defaultPostDeliveryConcurrency = 4`
- generic follow-on fanout currently uses
  `defaultPostFollowOnDeliveryConcurrency = 4`
- comments currently inherit that generic follow-on concurrency
- comments, post reactions, and comment reactions share the same durable
  engagement helper, so changing comment concurrency must not silently widen
  unrelated engagement behavior unless that is explicitly intended

## Locked Decisions

- fix receiver comment freshness by making the open comments surface refresh from
  Posts-owned persisted state, not by wiring the widget directly to raw
  transport messages
- prefer repository-driven refresh keyed by `postId` so the open sheet responds
  consistently to:
  - direct live transport
  - relay inbox drain
  - pending-child-event reconciliation
  - local sender actions
- keep `PostCommentListener` as the transport/persistence boundary
- raise post-create concurrency from `4` to `25`
- raise comment fanout concurrency from `4` to `25`
- do not silently widen reaction, comment-reaction, or pass-along concurrency in
  this plan; if needed, introduce event-type-aware engagement concurrency so
  `post_comment` can move independently
- do not change whether `deliveryStatus == 'inbox'` counts as settled in this
  plan

## Non-Goals

This plan does not cover:

- comment composer redesign
- new push or notification behavior
- redefining sender-side success semantics for comment delivery
- raising reaction or pass-along concurrency unless required as the smallest
  implementation consequence of isolating comment concurrency
- pin delivery changes

## Risks And Watchpoints

- a live-refreshing comments sheet can accidentally cause broad widget rebuilds
  or repeated async reload loops if it is not scoped tightly to the active post
- raising post-create concurrency to `25` may expose relay, bridge, or device
  bottlenecks hidden by the previous cap of `4`
- raising comment concurrency to `25` must not silently widen unrelated
  engagement event behavior without explicit tests
- this plan does not address the separate semantic issue where relay inbox store
  counts as sender-side settlement before the receiver has visually converged

## Acceptance Decision

Accept this slice when:

- an open receiver comments sheet live-updates from persisted state
- post-create bounded concurrency defaults to `25`
- comment bounded concurrency defaults to `25`
- existing comment ordering, dedupe, expiry refresh, and reconciliation
  semantics remain green
