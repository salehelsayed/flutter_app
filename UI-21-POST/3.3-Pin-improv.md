# Pin Follow-On Concurrency TDD

Prepared on: 2026-03-16
Prepared by: Codex
Depends on:
- `UI-21-POST/3.2-Post-reliability-and-follow-on-TDD.md`

## Goal

Reduce recipient-visible pin/update/remove latency by raising pin follow-on
fanout concurrency above the current serialized path.

This plan is intentionally narrow:
- improve `pinPost()`, `editPinnedPost()`, and `removePin()` fanout latency
- keep the durable outbox and stale-event protections intact
- do not widen scope into inbox-settlement semantics or non-pin follow-ons

## Current Findings

- pin follow-ons currently use `defaultPostPinDeliveryConcurrency = 1` in
  `lib/features/posts/application/post_pin_delivery_support.dart`
- pin recipients are loaded from `post_recipients` ordered by
  `recipient_peer_id`, so the current behavior is a serial queue
- each recipient can consume up to the interactive follow-on budget before the
  next recipient starts, which creates head-of-line blocking for larger friend
  sets
- durable retry and stale-event protection already exist and must remain intact

## Locked Decisions

- raise the default pin fanout concurrency to `25`
- keep the current pin-specific durable outbox architecture
- do not change the sender-visible settlement model in this plan
- do not change generic follow-on concurrency in this plan
- do not change stale-event ordering semantics in this plan

## Scope

In scope:
- `pinPost()`
- `editPinnedPost()`
- `removePin()`
- initial send and retry fanout for `post_pin_update` and `post_pin_remove`

Out of scope:
- redefining whether `inbox` counts as settled
- adding per-recipient UI for pin convergence
- changing comment/reaction/pass follow-on concurrency
- redesigning Posts UX

## Phase 2: Raise Pin Fanout Concurrency To 25

Outcome:
- pin update and pin remove fanout use bounded parallel delivery with a cap of
  `25`

### RED

Suggested test files:
- extend `test/features/posts/improvement/post_pin_delivery_support_test.dart`
- extend `test/features/posts/phase5/pin_post_use_case_test.dart`

Minimum RED cases:
- default pin follow-on fanout allows up to `25` in-flight recipients
- fanout never exceeds `25`
- later recipients can start before earlier slow recipients complete
- outbox persistence still records all recipient rows before delivery starts
- retry fanout also honors the new default cap

### GREEN

- change `defaultPostPinDeliveryConcurrency` from `1` to `25`
- ensure both initial send and retry continue to use the same default
- keep `interactivePostPinBudget` unchanged in this plan

### REFACTOR

- add a short code comment documenting why pin fanout intentionally diverges
  from the prior serialized policy

### Exit Gate

- pin/remove delivery is no longer serialized
- concurrency is bounded and deterministic at `25`

## Phase 3: Guard Correctness Under Higher Parallelism

Outcome:
- higher parallelism does not break authoritative pin behavior

### RED

Suggested test files:
- `test/features/posts/improvement/post_pin_retry_integration_test.dart`
- `test/features/posts/improvement/post_pin_remove_delivery_integration_test.dart`
- `test/features/posts/phase5/posts_wired_sender_pin_action_test.dart`

Minimum RED cases:
- a retried stale pin remove still cannot override a later re-pin
- partial settlement still reports `partiallySettled` instead of `success`
- fully unresolved remove still reports `queuedForRetry`
- sender local state still updates immediately while unresolved recipients stay
  retryable
- real sender -> router -> listener -> recipient-state pin remove still clears

### GREEN

- keep the existing stale-event comparison logic unchanged
- keep durable outbox retry selection unchanged
- verify higher concurrency does not reorder the final authoritative state on
  recipients

### REFACTOR

- keep concurrency mechanics isolated to pin delivery support, not UI code

### Exit Gate

- the latency improvement does not weaken authoritative pin correctness

## Risks And Watchpoints

- raising concurrency to `25` may increase simultaneous direct-send attempts and
  expose bridge or relay bottlenecks that were previously hidden by the
  serialized path
- if a sender has very large friend lists, `25` may still leave a visible tail,
  but it should remove the most obvious head-of-line blocking
- this plan does not address the separate semantic gap where relay inbox store
  is treated as settled before the recipient has visually converged

## Acceptance Decision

Accept this slice when:
- pin/update/remove fanout defaults to `25`
- bounded parallelism is locked by tests
- stale remove protection remains green
- the real recipient-state integration tests remain green
