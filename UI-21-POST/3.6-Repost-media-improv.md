Prepared on: 2026-03-16
Prepared by: Codex
Depends on:

## Goal

Fix the repost correctness and security gaps uncovered in the
Solz -> Hisam -> Ibra scenario:

- a repost recipient who was not in the original post audience must still be
  able to render reposted media correctly
- a repost recipient who does not know the original author as a contact must
  still see the original author's real avatar on the repost card
- repost delivery must no longer rely on the current plaintext follow-on path
  and should reuse the `post_create` delivery runner stack instead of keeping a
  second repost-only transport path

The desired outcome is:

- repost transport continues to use the existing libp2p secure transport path
- repost payloads are app-level encrypted per recipient like `post_create`
- repost media is independently retrievable for the new repost recipient set
- repost media blobs are encrypted before upload, so the relay stores ciphertext
  at rest rather than plaintext media
- repost cards are self-renderable for original-author identity, including the
  avatar, without depending on the receiver's contact list
- fresh repost recipients start with the correct historical engagement
  baseline:
  - existing heart state travels with the repost
  - existing repost total count travels with the repost
- reposted posts preserve one converged repost engagement thread:
  - comments, reactions, and comment reactions on the reposted post must fan
    out among the original author, the passer, and repost recipients who have
    actually engaged on that reposted post
  - repost delivery alone must not automatically subscribe every repost
    recipient to every later engagement from every other repost recipient
- later heart toggles and later repost events continue from that carried
  baseline instead of resetting the receiver copy to zero/one
- sender-side repost does not silently report success when it would produce a
  broken or unencryptable repost snapshot

## Current Findings

### Repost payload encryption gap

- `post_create` already uses a v2 encrypted envelope built with
  ML-KEM-768 + AES-256-GCM
- repost/pass-along currently builds a plaintext `post_pass` version `1`
  payload and sends that same string through the generic follow-on delivery path
- direct send and inbox fallback both operate on that plaintext repost string
- the current repost outbox stores one shared `raw_envelope`, which is
  incompatible with per-recipient encrypted payloads
- `post_create` already has a runner-based path for per-recipient wire-envelope
  building, recipient-delivery persistence, bounded concurrency, direct send,
  immediate inbox fallback, and durable retry; repost does not currently reuse
  that path

### Media contract gap

- original post media uploads are access-controlled to the original author plus
  the original recipient set
- repost/pass-along currently copies the original attachment metadata and the
  same `blob_id` values into the pass snapshot
- the repost receiver then tries to download those original `blob_id` values
- if the repost receiver was not in the original upload ACL, the download fails
  and the card renders broken media placeholders
- current tests partly codify the bug by expecting media repost to reuse the
  original `blob_id`

### Media at-rest encryption gap

- the current post media upload path uploads raw file bytes to the relay media
  store and relies on `allowedPeers` plus libp2p transport security
- that means the relay media store does not currently have a Posts-level
  guarantee that reposted media is encrypted at rest
- transport security alone is not sufficient for the repost requirement in this
  plan

### Avatar contract gap

- repost/pass-along currently carries original-author peer ID and username, but
  no self-renderable avatar snapshot
- receiver avatar rendering for posts relies on either caller-provided avatar
  bytes or a contact-avatar file already stored under
  `media/avatars/{peerId}.jpg`
- when the repost receiver does not have the original author as a contact, the
  repost card falls back to the deterministic ring/default avatar even though
  the repost correctly names the original author
- current incoming-pass handling validates the passer contact only; it does not
  create a Posts-owned avatar contract for the original author

### Shared-thread engagement gap

- post comments and reactions resolve recipients from persisted post recipient
  deliveries plus `authorPeerId`
- normal `post_create` receive persists those recipient deliveries, but repost
  receive does not currently persist any repost participant set
- repost/pass-along also does not currently carry a participant contract that
  would let receivers reconstruct the repost thread audience
- in the exact Solz -> Hisam -> Ibra scenario, that means Ibra commenting on
  the repost is likely to reach Solz as original author but miss Hisam as the
  passer
- the same continuity gap also affects comments or reactions initiated later by
  Hisam or Solz after the repost if the repost participant set is not durably
  represented

### Historical engagement baseline gap

- repost/pass-along does not currently carry any historical heart baseline for
  the post
- repost/pass-along does not currently carry any historical repost total
  baseline for the post
- fresh repost recipients therefore rebuild heart count only from their local
  `post_reactions` table and rebuild repost count only from their local
  repost/pass rows
- that means a fresh repost recipient typically starts at:
  - heart count `0`
  - repost count equal only to locally known repost rows, often just the
    incoming repost itself
- a scalar historical `heart_count` alone is not sufficient for correct live
  continuity because post hearts are sender-keyed toggle state:
  - if someone who was already hearted at repost time later unhearts, the
    receiver cannot reconcile that correctly from a count-only baseline
  - the repost contract therefore needs a stateful active-reaction baseline,
    not just one number

### Existing test coverage gap

- outgoing pass tests assert that media repost reuses the original `blob_id`
- incoming pass media tests use a fake hydrator and do not cover the
  unauthorized-recipient path
- no current repost test proves that repost send uses `message.encrypt`
- no current repost test proves that inbox fallback stores only encrypted repost
  payloads
- no current repost test proves that repost delivery/retry uses the same
  runner-style recipient-delivery path as `post_create`
- no current repost integration test covers the full encrypted direct-send,
  inbox-fallback, and retry-from-durable-state lifecycle for reposts
- no current repost test proves that relay-stored repost media is ciphertext
  rather than plaintext
- no current three-peer smoke regression locks the exact
  Solz -> Hisam -> Ibra case where the repost recipient is outside the original
  media ACL and does not know the original author as a contact
- no current repost test proves that a non-contact original author still needs
  to render with a real avatar on the receiver card

## Locked Decisions

- migrate repost delivery off the pass-specific follow-on outbox and onto the
  same runner/retry stack used by `post_create`
- repost delivery must satisfy three explicit security layers:
  - libp2p secure transport for the underlying stream
  - per-recipient app-level encrypted repost payloads using the same
    ML-KEM-768 + AES-256-GCM model as `post_create`
  - client-side encrypted repost media blobs so the relay stores ciphertext at
    rest
- retire plaintext repost send for the sender path:
  - sender-side repost must emit only encrypted repost payloads
  - receiver may keep the smallest compatibility read path for legacy plaintext
    reposts only if RED proves rollout compatibility is required
- reuse `PostDeliveryRunner`, `post_recipients`, aggregate delivery status, and
  `PendingPostDeliveryRetrier` for repost delivery wherever the runner can be
  extended safely
- if the runner needs the smallest extensibility seam to accept repost-specific
  wire-envelope building or repost-created local state, add that seam there
  instead of preserving `post_pass_follow_on_support.dart` as a second durable
  delivery path
- inbox fallback must store only encrypted repost payloads, and this plan
  reuses the runner's existing direct-send-then-inbox behavior rather than
  inventing a repost-only later inbox phase
- retire the repost-specific shared `raw_envelope` retry model in favor of
  rebuilding encrypted repost wire payloads from durable repost state plus
  recipient-delivery rows in the runner path
- for reposted media, build a repost-owned renderable snapshot before delivery:
  - repost media must use new `blob_id` values owned by the repost send
  - the repost media ACL must be derived from the resolved repost recipient set,
    not from the original post audience
- repost media bytes must be encrypted locally before upload with a repost-owned
  symmetric content key per attachment
- the recipient must decrypt repost media locally after download using key
  material obtained from the decrypted repost payload
- the relay must store only ciphertext repost media bytes; do not add a
  relay-side plaintext exception for repost blobs
- preserve media order and renderability across images, video, and voice media
- if the sender cannot prepare a fully renderable and encryptable repost media
  snapshot from locally available bytes, do not send a broken repost:
  - fail explicitly on the sender path
  - do not queue a repost payload that points at inaccessible or unencryptable
    media
- keep one-hop limit, `pickPeople` rejection, explicit-recipient selection, and
  original-author notification intact
- make reposts self-renderable for original-author identity by carrying a small
  avatar snapshot inside the encrypted repost payload
- persist that avatar snapshot in Posts-owned durable state keyed by the repost
  post, not in the contacts feature and not via a synthetic contact
- the feed card must prefer the Posts-owned avatar snapshot for passed-along
  posts when present, and fall back gracefully when absent
- reposted posts must preserve one converged engagement thread, but the fanout
  topology is intentionally narrower than "all repost recipients hear
  everything":
  - the original author always participates
  - the passer always participates
  - a repost recipient becomes a repost-engagement participant only after that
    recipient actually engages on the reposted post
- the encrypted repost payload must also carry a historical engagement baseline
  for the reposted post
- for hearts, that baseline must be stateful active-reaction data keyed
  strongly enough that later reaction toggles can be reconciled correctly after
  the repost:
  - do not rely on a scalar `heart_count` alone
  - carry the minimum hidden reactor-state snapshot needed for correct future
    like/unlike convergence
- incoming repost receive must persist that hidden heart baseline into
  Posts-owned durable state before later reaction events are applied
- for repost totals, the encrypted repost payload must carry the current shared
  repost metric baseline so a fresh repost recipient can render the existing
  total immediately
- later repost events must continue from that carried repost baseline on copies
  that surface repost totals, rather than restarting from locally observed
  repost rows only
- add one Posts-owned repost engagement-participant contract so the repost
  thread audience can be reconstructed durably on sender and receiver devices
- the encrypted repost payload must carry the minimum passer/origin metadata
  needed to rebuild that repost engagement topology on receipt
- sender-side local repost creation and incoming repost receive must both
  persist the base repost-engagement participants into Posts-owned durable
  state:
  - original author
  - passer
- when a repost recipient hearts, comments, or comment-reacts on that reposted
  post, persist that recipient durably as a repost-engagement participant for
  future continuity
- comment, reaction, and comment-reaction recipient resolution for repost-thread
  continuity must union:
  - original author
  - passer
  - repost-engagement participants created by prior repost engagement
  while still excluding the current sender
- do not automatically widen repost engagement fanout to every repost-delivery
  recipient or every original post recipient
- keep blocked/archived contact filtering unchanged when expanding repost
  participant fanout
- preserve orphan child-event staging and replay for repost engagement:
  - if a repost-linked comment arrives before the reposted post exists locally,
    stage it safely
  - replay it only after the reposted post plus repost-engagement baseline
    state have both been persisted
- do not redesign the Posts UI in this plan
- do not change sender settlement semantics in this plan
- do not widen scope into the separate `3.5` sender-UX/concurrency work except
  for the smallest compatibility change required by encrypted repost delivery

## Non-Goals

This plan does not cover:

- redesigning repost UX, card layout, or sender flow timing
- raising repost fanout concurrency
- adding a generic cross-feature non-contact avatar system
- adding a relay-side blob-clone or ACL-mutation feature for reposts
- retrofitting app-level encrypted follow-on delivery for comments, reactions,
  pins, or other non-repost events
- redesigning the generic comment/reaction transport architecture outside the
  repost-engagement continuity needed by this plan
- carrying the full historical comment thread inside reposts; this plan only
  adds the historical heart and repost metric baselines plus future
  shared-thread continuity
- changing the post-create runner's inbox timing policy; this plan reuses its
  current direct-then-inbox behavior
- guaranteeing repost of media that the sender has never downloaded locally;
  this plan prefers explicit sender failure over silently forwarding an
  unretrievable snapshot

## Risks And Watchpoints

- recipient-specific encrypted repost payloads increase CPU work and make the
  current repost follow-on helper obsolete; the runner reuse must keep retry
  semantics durable without regressing `post_create`
- client-side media encryption introduces temporary ciphertext files and key
  material that must be cleaned up correctly after upload or failure
- repost-owned media uploads will increase sender-side work and data usage; the
  plan accepts that cost to restore correctness and confidentiality for new
  recipients
- if the sender does not have local bytes for a reposted attachment yet, this
  plan prefers explicit sender failure over forwarding an inaccessible snapshot
- adding avatar snapshot data and media key material to the repost contract
  increases payload size; keep both bounded and compact
- carrying repost participant continuity data in the encrypted payload and
  durable Posts state increases repost contract size and query complexity; keep
  the participant representation minimal and explicit
- carrying hidden active-heart baseline state in the encrypted payload and
  durable Posts state increases repost contract size; keep that state minimal,
  explicit, and internal to Posts rather than turning it into a general social
  graph feature
- a scalar carried heart total would be smaller but incorrect for future
  unlike/re-like continuity; this plan accepts the extra state needed for
  correctness
- extending `PostDeliveryRunner` for repost-specific envelope building must not
  regress plain `post_create` encryption, fanout, or retry semantics
- persisting avatar snapshot bytes or repost crypto metadata in Posts-owned
  state must not bloat every feed load unexpectedly; store only what the UI and
  retry path need
- the repost fix must not overwrite or interfere with the original author's
  existing contact/avatar data when the receiver does know that author
- if rollout compatibility requires temporary support for legacy plaintext
  repost receive, that compatibility path must stay receive-only and not reopen
  plaintext send for new reposts
- this plan intentionally does not solve the separate repost sender-UX timing
  and concurrency concerns covered in `3.5`

## Acceptance Decision

Accept this slice when:

- a repost recipient outside the original post audience can render reposted
  media without broken placeholders
- a repost recipient who is not a contact of the original author can render the
  original author's actual avatar on the repost card
- repost payloads are app-level encrypted like `post_create`
- relay-stored repost media is encrypted at rest rather than plaintext
- reposted posts preserve the intended repost-thread topology:
  the original author, the passer, and repost recipients who have actually
  engaged on that reposted post
- reposted posts preserve the historical heart baseline and historical repost
  total baseline needed for a fresh recipient to render the current post state
  immediately
- repost delivery/retry reuses the `post_create` runner stack rather than the
  old follow-on helper path
- sender-side repost fails explicitly instead of silently forwarding an
  unretrievable or unencryptable repost snapshot
- one-hop limits, `pickPeople` rejection, durable retry, and idempotent pass
  handling remain green
