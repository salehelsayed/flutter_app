## Goal

Close the remaining sender-side reliability gaps after `3.1-Post-improv`:
- restart-safe media upload recovery for optimistic media posts
- durable retry for outgoing follow-on Post events
- a real fix for the pin-remove regression where recipients can stay pinned
- consistent handling for comments, reactions, comment reactions, and repost
  (`passPostAlong()`) under the new send architecture

This plan is reliability work. It does not redesign Posts UX or extract a
cross-feature shared helper.

## Current Findings

### What already exists

- post-create recipient delivery now has a local-create boundary, background
  runner, bounded fanout, and reconnect retry
- follow-on Post events now share one Posts-owned transport helper:
  - `send_post_reaction_use_case.dart`
  - `send_post_comment_use_case.dart`
  - `send_post_comment_reaction_use_case.dart`
  - `pass_post_along_use_case.dart`
  - `post_pin_delivery_support.dart`
  - `publish_post_presence_update_use_case.dart`

### What is still missing

- media upload recovery is session-only; media drafts are not persisted
- restart retry skips media posts whose upload stage never completed
- follow-on delivery returns one `bool` and does not persist per-recipient
  state
- comments and reactions are only saved locally after send succeeds, so sender
  intent is dropped on failure
- pass-along uses the shared helper but is not on a durable queued or retryable
  path
- pin update and pin remove use the shared helper, but success semantics are
  too weak for authoritative state changes
- `PostsWired` currently ignores the `removePin(...)` result and reloads as if
  the remove succeeded

### Terminology lock

- In this codebase, "repost" means `passPostAlong()`.
- Comments, reactions, comment reactions, repost/pass-along, pin update, and
  pin remove are all "follow-on Post events" for this plan.

## Locked Decisions

### Architecture

- Do not reuse `PostPendingChildEvent` for outgoing retry. That model is for
  inbound orphan staging, not sender-side delivery recovery.
- Add one Posts-owned outgoing follow-on outbox with per-event and
  per-recipient state.
- Add one Posts-owned outgoing media-upload recovery contract keyed by post and
  media position.
- Keep post-create retry separate from follow-on retry, but allow media-recovery
  completion to hand off into the existing post-create delivery runner.
- Do not extract a cross-feature helper in this plan.

### Scope

- Include comments, reactions, comment reactions, and repost/pass-along in the
  durable follow-on architecture.
- Include pin update and pin remove in that same architecture because pin state
  is authoritative and currently broken.
- Keep nearby presence out of scope for this TDD unless the generic outbox work
  makes it almost free as a later follow-up. Presence is not the current
  blocking defect.

### Sender contract

- A sender-side pin remove must not silently report success when recipients have
  not converged.
- Follow-on sends need more than `bool deliveredAny`; the sender path must know
  whether delivery is:
  - fully settled
  - partially settled
  - not settled
- Before durable retry lands, the pin/remove flow must at minimum expose a
  failure state instead of silently accepting divergence.

### Media recovery contract

- Optimistic media posts must survive app kill between:
  - local post creation
  - media preparation
  - media upload
  - recipient fanout
- A media post must never become a permanent dead-end skeleton just because the
  app restarted before upload completed.

## Non-Goals

This plan does not cover:
- chat transport extraction or chat semantic changes
- OS background workers
- a new feed-card design or new feed ordering rules
- new per-recipient delivery UI for follow-on events
- a full nearby presence retry rollout

## Phase 1: Reproduce And Lock The Pin-Remove Regression

Outcome:
- the current bug is reproduced by an end-to-end sender -> router -> listener
  -> recipient-state test
- the desired sender behavior on failed pin remove is explicit before broader
  retry work starts

### RED

Suggested test files:
- `test/features/posts/improvement/post_pin_remove_delivery_integration_test.dart`
- extend `test/features/posts/phase5/pin_post_use_case_test.dart`
- extend `test/features/posts/phase5/posts_wired_sender_pin_action_test.dart`

Minimum RED cases:
- author pins a post and the recipient becomes pinned through the real transport
  and listener path
- author removes the pin and the recipient pinned section disappears through the
  real transport and listener path
- if direct send and inbox storage both fail, sender and recipient do not
  silently appear converged
- `PostsWired` does not ignore a failed `removePin(...)` result

### GREEN

- add the missing sender/recipient integration harness
- make `PostsWired._removePinnedPost()` inspect `RemovePinResult`
- lock one temporary sender behavior before durable retry lands:
  - either rollback the local remove on full failure
  - or show an explicit failed/pending-removal state and do not treat it as a
    clean success

### REFACTOR

- isolate pin-remove orchestration from generic sheet reload logic
- keep the transport-path reproduction harness reusable for later pin retry work

### Exit Gate

- the reported pin-remove regression is reproduced first
- the sender UI no longer silently accepts a failed remove

## Phase 2: Add A Durable Outgoing Follow-On Outbox

Outcome:
- follow-on Post events have persisted sender-side jobs and persisted
  per-recipient delivery rows

### RED

Suggested test files:
- `test/core/database/migrations/034_posts_follow_on_outbox_test.dart`
- `test/features/posts/improvement/post_follow_on_outbox_repository_test.dart`
- extend `test/features/posts/improvement/post_follow_on_delivery_test.dart`

Minimum RED cases:
- migration creates one event table and one recipient-delivery table for
  outgoing follow-on events
- repository round-trip preserves:
  - event id
  - event type
  - raw envelope
  - parent ids such as `post_id` and `comment_id` when present
  - sender id
  - created-at timestamp
  - recipient peer ids
  - per-recipient delivery status, path, last error, and attempt timestamps
- loading retryable follow-on jobs excludes already-settled recipients
- the delivery helper returns a structured result instead of one boolean
- concurrency tests still prove the cap and lazy slot opening

### GREEN

- add Posts-owned outgoing follow-on models and repository APIs
- add the DB migration and helpers
- change the follow-on delivery helper to return per-recipient outcomes and an
  aggregate result
- keep the helper Posts-owned and feature-scoped

### REFACTOR

- isolate scheduling from business-state aggregation
- keep raw envelope persistence separate from per-recipient state updates

### Exit Gate

- follow-on send recovery is backed by persisted state, not an in-memory bool
- the code can distinguish full success, partial success, and failure

## Phase 3: Put Pin Update And Pin Remove On The Durable Outbox

Outcome:
- pin update and pin remove become restart-safe authoritative sender events
- the current unpin bug is fixed with durable recovery, not just best-effort UI

### RED

Suggested test files:
- extend `test/features/posts/improvement/post_pin_remove_delivery_integration_test.dart`
- `test/features/posts/improvement/post_pin_retry_integration_test.dart`
- extend `test/features/posts/improvement/post_pin_delivery_support_test.dart`
- extend `test/features/posts/phase5/posts_wired_pinned_section_test.dart`

Minimum RED cases:
- pin update queues and sends through the outbox without losing current pin
  semantics
- pin remove does not report clean success when only some recipients settle
- unresolved pin-remove recipients retry on reconnect or restart
- after recovery, the recipient pinned section disappears without direct repo
  mutation in the test
- sender removal still updates the regular feed entry correctly per
  `Posts-Feed-Rules.md`
- repeated pin/remove cycles do not duplicate state or regress the latest-event
  wins rule

### GREEN

- move `pinPost()`, `editPinnedPost()`, and `removePin()` onto the new outbox
- queue the event before network delivery
- retry unresolved recipients through a Posts-owned follow-on retrier
- make sender-side result handling explicit in `PostsWired`

### REFACTOR

- reuse one common follow-on enqueue-and-deliver path for pins
- keep pin-specific rule checks separate from generic outbox mechanics

### Exit Gate

- the user-reported unpin bug is covered by an end-to-end regression test
- authoritative pin events no longer depend on `bool deliveredAny`

Minimum RED cases:

  dropping the sender action

- keep engagement-specific dedupe and idempotency rules outside the generic

### Exit Gate

- sender comments and reactions are no longer lost on transient send failure
- engagement flows now follow the durable send architecture, not best-effort

## Phase 5: Move Repost/Pass-Along Onto The New Path

Outcome:

### Exit Gate

## Phase 6: Persist Outgoing Media Upload Recovery State

Outcome:
- optimistic media posts have a restart-safe upload contract

### RED

  - duration and waveform when present
  - position order

- a post can be discovered as pending media upload after app restart

- save draft upload metadata during or immediately after local post creation

- keep draft recovery storage minimal but sufficient

### Exit Gate

- media posts can survive app kill before upload completes
- the code no longer relies on in-memory `mediaDrafts` for restart recovery

## Phase 7: Add A Media Upload Retrier And Handoff To Post-Create Retry

- restart-safe media recovery completes upload first, then resumes recipient

### RED

## Acceptance Decision

This TDD should be treated as the next active Posts reliability plan after
`3.1-Post-improv`.

Recommended execution order:
1. Phase 1 and Phase 2 first, because the current pin-remove regression needs a
   real sender-side contract and a durable foundation.
2. Phase 3 next to fix authoritative pin convergence.
3. Phase 4 and Phase 5 next to bring comments, reactions, and repost/pass-along
   onto the same reliable architecture.
4. Phase 6 and Phase 7 after that to close the deferred media restart recovery
   gap without mixing it into the follow-on outbox work.
