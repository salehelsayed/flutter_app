## Investigation Summary

### Why Phase 4 Is Not Blocked By Phases 1-3

Phases 1-3 are fully complete and verified (32/32 tests passing):
- Phase 1: 4 smoke tests lock the broken media/avatar/plaintext baseline
- Phase 2: Per-recipient ML-KEM-768 + AES-256-GCM encrypted repost payloads via
  `PostDeliveryRunner.executePostPass()` — direct send, inbox fallback, retry all
  proven
- Phase 3: Repost engagement participants, heart baselines, and repost-total
  baselines persisted via migration 037 + `post_repost_engagement_support.dart`

### The 5 Concrete Gaps Phase 4 Must Close

| # | Gap | Layer | Current State |
|---|-----|-------|---------------|
| 1 | No Go-side file-level encryption | `go-mknoon/crypto/`, `go-mknoon/bridge/bridge.go` | Only string-based `message.encrypt`/`message.decrypt` exist; no `blob:encrypt`/`blob:decrypt` for large files |
| 2 | No repost media crypto metadata in DB | `post_media_attachments` table | No columns for encryption key, nonce, algorithm, or encrypted flag |
| 3 | No repost media preparation stage | `pass_post_along_use_case.dart` lines 119-190 | Copies original `blob_id` values via `toRenderableJson()` — new recipients outside the original ACL get 403 |
| 4 | No decrypt-after-download step | `download_post_media_use_case.dart` lines 22-65 | Writes raw bytes directly to disk; no crypto metadata check |
| 5 | `PostPassEnvelope` doesn't carry media key material | `post_pass_envelope.dart` `_buildPayloadJson()` lines 338-370 | Serializes `attachment.toRenderableJson()` with original `blob_id`, no per-attachment content key |

### Current Media Pipeline (Repost Path)

```
passPostAlong()
  → loads post.media (original attachments with original blob_ids)
  → validates via isValidSnapshotMedia()
  → PostPassEnvelope.fromPass(media: post.media)
      → _buildPayloadJson() → attachment.toRenderableJson()
          → serializes original blob_id, kind, mime, size, etc.
  → deliverCreatedLocalPostPass()
      → PostDeliveryRunner.executePostPass()
          → _buildPostPassWireEnvelope() per recipient (ML-KEM encrypted)
  → recipient receives, decrypts envelope, parses media with original blob_ids
  → callP2PMediaDownload(id: originalBlobId) → 403 if outside original ACL
```

### Target Media Pipeline (After Phase 4)

```
passPostAlong()
  → loads post.media (original attachments)
  → validates via isValidSnapshotMedia()
  → VERIFY local bytes exist for each attachment (fail explicitly if not)
  → FOR EACH attachment:
      → generate repost-owned symmetric content key (AES-256-GCM)
      → callBlobEncrypt(filePath: localBytes, key: contentKey) → ciphertextPath + nonce
      → generate new blob_id for ciphertext
      → callP2PMediaUpload(id: newBlobId, filePath: ciphertextPath,
          allowedPeers: [passer, ...repostRecipients, originalAuthor])
      → create repost attachment model with new blob_id + crypto metadata
  → PostPassEnvelope.fromPass(media: repostAttachments, mediaKeys: keyMap)
      → _buildPayloadJson() → includes per-attachment content key + nonce
  → deliverCreatedLocalPostPass() → encrypt per recipient → deliver

recipient:
  → decrypt envelope → extract media key material per attachment
  → persist repost attachment rows with crypto metadata
  → callP2PMediaDownload(id: newBlobId, outputPath: ciphertextPath)
  → callBlobDecrypt(filePath: ciphertextPath, key: contentKey, nonce: nonce)
      → plaintextPath
  → update attachment localPath + downloadStatus: 'done'
```

New helper (private or extracted):
```dart
Future<_RepostMediaPrepResult?> _prepareRepostMedia({
  required Bridge bridge,
  required List<PostMediaAttachmentModel> originalMedia,
  required String passerPeerId,
  required List<String> recipientPeerIds,
  required String originalAuthorPeerId,
}) async {
  final repostAcl = <String>{
    passerPeerId,
    ...recipientPeerIds,
    originalAuthorPeerId,
  }.toList(growable: false);

  final attachments = <PostMediaAttachmentModel>[];
  final keys = <String, PostMediaCryptoEntry>{};

  for (final attachment in originalMedia) {
    // 1. Verify local bytes exist
    if (attachment.localPath == null || !File(attachment.localPath!).existsSync()) {
      return null; // Explicit failure
    }

    // 2. Generate content key
    final keyBase64 = await callBlobKeygen(bridge);

    // 3. Encrypt locally
    final (:encryptedPath, :nonce) = await callBlobEncrypt(
      bridge,
      filePath: attachment.localPath!,
      keyBase64: keyBase64,
    );

    // 4. Generate new blob_id
    final newBlobId = 'blob_${uuid.v4()}';

    // 5. Upload ciphertext with repost ACL
    await callP2PMediaUpload(
      bridge,
      id: newBlobId,
      toPeerId: passerPeerId,
      mime: attachment.mime,
      filePath: encryptedPath,
      allowedPeers: repostAcl,
    );

    // 6. Build repost attachment
    attachments.add(attachment.copyWith(
      blobId: newBlobId,
      encryptionKeyBase64: keyBase64,
      encryptionNonce: nonce,
      isEncrypted: true,
    ));

    keys[attachment.mediaId] = PostMediaCryptoEntry(
      keyBase64: keyBase64,
      nonce: nonce,
      blobId: newBlobId,
    );

    // 7. Clean up temp encrypted file
    File(encryptedPath).deleteSync();
  }

  return _RepostMediaPrepResult(attachments: attachments, keys: keys);
}
```

## Risks And Watchpoints

- **Memory pressure from in-memory GCM:** AES-256-GCM `Seal()`/`Open()` loads
  the entire file into memory. For typical post media (< 50 MB) this is
  acceptable. If post media limits grow, switch to streaming
  AEAD (e.g., `miscreant` or chunked GCM).

- **Temporary ciphertext file cleanup:** The sender creates temp `.enc` files
  during media preparation. These must be cleaned up on both success (after
  upload) and failure (on any error). Use a `try/finally` pattern.

- **Upload timeout:** Media upload has a 5-minute timeout. Encrypting + uploading
  multiple large attachments sequentially could approach this. Consider whether
  parallel preparation is needed (not in this phase — keep sequential for
  simplicity).

- **Retry must not re-upload:** The `inner_payload_json` persisted in the DB
  already contains the `media_keys` map with the new `blob_id` values. The
  delivery runner must detect that media is already prepared and skip the
  preparation stage on retry. The cleanest signal is: if `inner_payload_json`
  is non-null in the loaded pass, media preparation already ran.

- **Thumbnail blobs:** If the original post has `thumbnailBlobId`, the repost
  media preparation must also re-upload the thumbnail with a new blob_id and
  encrypted bytes. Keep the same encrypt-upload-cleanup pattern per thumbnail.

- **Post-create media path unchanged:** Phase 4 must not change the original
  `attach_post_media_use_case.dart` upload path. Post-create media is a
  separate concern — only repost media gets blob-encrypted at rest.

- **Legacy plaintext reposts:** Receiver must still handle legacy v1 reposts
  with unencrypted media (no `media_keys` in payload). The download path
  already handles `isEncrypted == false` gracefully.

## Acceptance Decision

Accept this phase when:

- `passPostAlong()` for media posts generates new repost-owned `blob_id` values
- Repost media bytes are encrypted locally before upload (relay stores ciphertext)
- Repost media ACL is derived from the repost recipient set, not the original
- A receiver outside the original ACL can download and decrypt repost media
- Sender-side repost fails explicitly if local media bytes are missing or
  encryption fails
- Retry reuses prepared encrypted media without duplicate uploads
- Post-create media upload path is unchanged
- Three-peer smoke test passes end-to-end with encrypted repost media
