# Repost/Pass-Along UX And Delivery TDD

Prepared on: 2026-03-16
Prepared by: Codex
Depends on:
- `UI-21-POST/3.2-Post-reliability-and-follow-on-TDD.md`

## Goal

Make repost/pass-along feel fast and visible again without regressing the
durability and retry guarantees introduced in `3.2`.

This plan targets three concrete problems:

- sender UX is slower than post create because the pass sheet waits for network
  delivery before closing
- receiver UX is muted when a pass arrives for a post that already exists
  locally, so the event can land without producing an obvious visible update
- repost delivery still uses the generic follow-on concurrency of `4` even
  though post create now uses bounded concurrency `25`

The desired outcome is:

- sender-side repost feels local-first like post create
- recipient-side repost is visibly surfaced even when the original post already
  exists
- repost fanout and repost retry use bounded concurrency `25`

## Current Findings

### Sender path

- `passPostAlong()` already persists a local `PostPassModel` and durable outbox
  recipient rows before delivery completes
- recent Posts work now uses an explicit local-persist-then-background-deliver
  split for comments, so repost no longer needs to invent a bespoke widget-side
  orchestration pattern
- `PostsWired._passAlong()` does not exploit that local-first persistence
  boundary
- `PassPostAlongSheet` awaits the full `onSubmit(...)` future and only then
  dismisses, so the sender waits on network and inbox fallback before the sheet
  closes
- unlike post create, repost does not currently split local persistence from
  background delivery

### Receiver path

- repost is routed and persisted through the new router -> listener -> use-case
  path, not the legacy fire-and-forget path
- if the receiver does not already have the post, the incoming pass creates a
  visible passed-along card
- if the receiver already has the original post, `handleIncomingPassedPost()`
  currently persists the `post_pass` record but intentionally avoids turning the
  existing direct post into a visibly passed-along card
- receiver cards also hide share counts for non-authors, so an already-known
  post can receive a pass and still appear visually unchanged

### Delivery path

- repost still uses the pass-specific follow-on outbox and retry path
- repost fanout currently inherits `defaultPostFollowOnDeliveryConcurrency = 4`
- repost retry also inherits that same generic follow-on cap
- post create already uses `PostDeliveryRunner`, `post_recipients`,
  `PendingPostDeliveryRetrier`, and bounded concurrency `25`
- repost now visibly lags behind the current post-create baseline and also
  carries a second delivery/retry stack that overlaps heavily with what
  post-create already has

## Locked Decisions

- move repost delivery off the pass-specific follow-on outbox and onto the same
  delivery/retry stack used by `post_create`
- reuse `PostDeliveryRunner`, `post_recipients`, aggregate delivery status, and
  `PendingPostDeliveryRetrier` for repost delivery where the runner can be
  extended safely
- introduce a repost-owned local-persist-then-background-deliver orchestration
  so sender UX mirrors post create at the surface level
- mirror the current Posts pattern used by comment send:
  - one create-local repost step that persists the pass and the recipient-
    delivery state needed by the reused runner path
  - one deliver-created repost step that performs network fanout later through
    the reused post delivery runner stack
  - keep `passPostAlong()` as an optional wrapper only if that materially
    simplifies compatibility for existing call sites
- if the reused runner needs the smallest extension to accept repost-specific
  wire-envelope building, make that extension in the runner path rather than
  preserving a separate repost follow-on transport stack
- keep one-hop limit, `pickPeople` rejection, explicit-recipient selection,
  original-author notification, and renderable snapshot validation intact
- if the sender tries to select the original author explicitly, filter that
  author out of the explicit repost audience:
  - the author may still receive the implicit author-notification copy
  - that author notification must not masquerade as an explicit repost
    recipient
- when a repost arrives for a receiver who already has the original post:
  - do not create a duplicate post row
  - do update persisted surfacing so the pass is visible on the existing card
  - prefer persisted pass attribution plus repost-time resurfacing over a
    transport-only transient signal
- when a direct author copy later arrives for a post that already exists locally
  as a reposted copy:
  - merge by `postId`
  - do not create a duplicate card
  - prefer the direct author relationship for stored sender/origin semantics
    while preserving any relevant pass history needed for repost surfacing
- preserve the existing rule that share counts remain an author-only metric
  unless the smallest viable implementation absolutely requires revisiting that
- introduce repost-specific bounded concurrency at `25`
- reuse the post-create runner's existing direct-send-then-inbox fallback
  behavior for repost delivery; do not invent a repost-only offline inbox phase
  in this plan
- do not silently widen comment, reaction, comment-reaction, or pin concurrency
  in this plan
- do not redefine whether `deliveryStatus == 'inbox'` counts as settled in this
  plan

## Non-Goals

This plan does not cover:

- redesigning the pass-along UI beyond the smallest UX changes required to make
  repost feel immediate and visible
- changing sender settlement semantics or adding cross-feature delivery receipts
- changing chat or non-Posts transport semantics
- changing generic follow-on concurrency for non-repost event types
- changing pin, comment, reaction, or media-upload behavior except for the
  smallest compatibility fix required by repost work
- solving repost participant continuity for comments/reactions across all repost
  participants; that belongs to `3.6`
- changing the post-create runner's inbox timing policy; this plan reuses the
  existing direct-then-inbox fallback behavior

  - reason: Phase 3 changes persisted repost surfacing semantics for already-

- Phase 4: `flutter-feature-module-implementer`,

  `flutter-test-orchestrator`
  - reason: Phase 4 moves repost initial-send and retry fully onto the

## Phase 2: Split Local Repost Persistence From Background Delivery

Outcome:
- sender repost behaves local-first like post create: local state is persisted,
  the sheet closes, and delivery continues in the background

### RED

Suggested test files:
- extend `test/features/posts/phase4/posts_wired_pass_along_test.dart`
- extend `test/features/posts/phase4/pass_post_along_use_case_test.dart`
- optionally add `test/features/posts/improvement/post_pass_sender_ux_test.dart`

Minimum RED cases:
- after tapping `Send pass`, the sheet closes once local pass and repost
  recipient-delivery state are persisted, even if direct send is still blocked
- sender-local repost state is visible immediately after dismissal through the
  existing repository-driven refresh path
- unresolved delivery still remains durable and retryable after the local-first
  UI returns control
- a full delivery failure does not erase the local pass record or queued
  recipient-delivery rows just because the sheet already dismissed
- the reused repost delivery path still routes through the post-create runner
  stack rather than the old repost follow-on helper

### GREEN

- split local repost creation/queueing from the delivery attempt
- introduce an explicit repost create-local step and repost deliver-created step
  so the sender path matches the current local-first comment architecture
- hand repost delivery to the reused post-create delivery runner path after the
  local repost state is persisted
- keep `passPostAlong()` semantics available only if a wrapper materially helps
  preserve compatibility for existing call sites
- make `PassPostAlongSheet` close after the local queue step instead of waiting
  for network settlement
- launch repost delivery in the background from Posts UI orchestration, not from
  the sheet widget itself

### REFACTOR

- keep repost local-queue orchestration out of the widget tree and in Posts
  application/UI coordination code
- mirror the new comment local-first split closely enough that future
  reliability work can reason about post create, comments, and repost using the
  same shape

### Exit Gate

- repost no longer blocks the sender on network delivery before the sheet closes
- sender-local repost state survives failed or delayed delivery

## Phase 3: Make Receiver-Targeted Reposts Visibly Surface Existing Posts

Outcome:
- a receiver who already has the original post still gets an obvious visible
  update when someone passes that post along to them

### RED

Suggested test files:
- extend `test/features/posts/phase4/handle_incoming_passed_post_use_case_test.dart`
- extend `test/features/posts/phase4/post_card_passed_along_test.dart`
- extend `test/features/posts/phase4/posts_wired_share_count_test.dart`
- extend `test/features/posts/phase1/handle_incoming_post_use_case_test.dart`
- optionally add `test/features/posts/improvement/post_pass_receiver_surface_test.dart`
- add `test/features/posts/improvement/post_pass_direct_merge_integration_test.dart`

Minimum RED cases:
- when a non-author receiver already has the direct original post and receives a
  repost, the existing post is resurfaced visibly from persisted state instead
  of staying visually unchanged
- that resurfacing does not create a duplicate post row
- persisted pass attribution is available to the UI for that resurfaced card
- feed ordering uses repost-time visibility for the receiver-facing resurfacing
  instead of leaving the card buried at the original post timestamp
- if the device already has the reposted copy and later receives the direct
  author copy for the same `postId`, the merge remains duplicate-free and does
  not leave two visible cards behind
- that later direct-copy merge can update the stored direct relationship
  without erasing repost attribution needed for receiver surfacing rules
- original-author share-count behavior remains intact
- duplicate deliveries of the same `passId` remain idempotent
- one fake-network integration must prove the end-to-end merge order that unit
  tests alone can miss:
  - receiver first gets the reposted copy
  - receiver later gets the direct author copy for the same `postId`
  - the feed still shows one card
  - the stored relationship upgrades without duplicate surfacing

### GREEN

- update incoming-pass persistence so receiver-targeted reposts can refresh the
  existing post’s surfaced state from durable data
- persist the minimum additional metadata needed for the UI to render
  passed-along attribution on an already-known post
- refresh the receiver-facing card’s surfaced ordering using the pass timestamp
  without creating a second visible post entry
- preserve the original post author/content identity while surfacing the passer
  attribution
- add the smallest direct-post compatibility merge so a later direct author
  copy upgrades the relationship without introducing a duplicate feed row

### REFACTOR

- keep receiver surfacing derived from persisted Posts state rather than from a
  transport-only stream callback
- if distinguishing author-notification behavior from recipient-facing behavior
  requires own-peer context, inject that explicitly instead of burying it in UI
  assumptions

### Exit Gate

- repost arrival is visibly surfaced for recipients even when the original post
  already exists locally
- no duplicate cards are introduced

## Phase 4: Raise Repost Fanout Concurrency To 25

Outcome:
- repost initial send and repost retry reuse the post-create delivery runner's
  bounded parallel delivery with a cap of `25`

### RED

Suggested test files:
- extend `test/features/posts/improvement/post_delivery_runner_parallel_test.dart`

- extend `test/features/posts/improvement/post_delivery_retry_integration_test.dart`
- extend `test/core/services/pending_post_delivery_retrier_test.dart`

Minimum RED cases:
- repost initial fanout allows up to `25` in-flight recipients
- repost fanout never exceeds `25`
- later repost recipients can start before earlier slow recipients finish
- repost retry also uses the same `25` cap through the post delivery retrier
- comment/reaction follow-ons do not inherit the repost cap accidentally
- repost delivery no longer depends on the pass-specific follow-on retrier path

### GREEN

- reuse the post-create runner's default concurrency cap of `25` for repost
- route repost retry through `PendingPostDeliveryRetrier` / recipient-delivery
  state instead of `PendingPostFollowOnRetrier`
- keep generic follow-on defaults unchanged for non-repost event types
- keep the runner's interactive direct-send budget unchanged in this plan unless
  RED proves the bottleneck is still dominated by the current budget after the
  transport-stack reuse

### REFACTOR

- make repost delivery policy explicit in the reused runner path instead of
  leaving repost on an accidental inherited follow-on default

### Exit Gate

- repost fanout is bounded at `25`
- repost retry uses the same runner-based retry stack as post create
- unrelated follow-on event types keep their existing concurrency policies

## Risks And Watchpoints

- closing the pass sheet after local queueing can make sender UX feel faster but
  also makes it easier to hide unresolved delivery unless the persisted state
  remains inspectable and retryable
- resurfacing existing posts on incoming repost may reorder the feed more
  aggressively than today; that is intentional, but the scope must stay limited
  to repost-targeted visible updates
- if author-notification behavior needs to remain different from recipient
  behavior, introducing own-peer context into pass handling must not create
  startup-order or dependency tangles
- reusing `PostDeliveryRunner` for repost may require a small extensibility seam
  so repost can build its own wire envelope without pretending to be a plain
  `post_create`
- reusing the runner also means repost inherits the runner's direct-then-inbox
  fallback semantics; this plan does not defer inbox until a later offline-only
  phase
- this plan intentionally does not solve the separate sender-semantic issue that
  relay inbox store counts as settled before the receiver has visually converged

## Acceptance Decision

Accept this slice when:

- sender repost closes quickly after local persistence and continues delivery in
  the background
- receiver-targeted reposts visibly resurface existing posts without creating
  duplicate cards
- repost fanout and repost retry reuse the post-create delivery runner stack at
  bounded concurrency `25`
- one-hop rules, snapshot validation, outbox durability, and idempotent retry
  behavior remain green
