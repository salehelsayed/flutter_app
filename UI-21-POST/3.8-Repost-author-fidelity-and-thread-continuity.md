## Why This Needs A New Plan

The current repost work is split across:

- `3.5` for repost sender and fanout behavior
- `3.6` for repost media, avatar, encryption, and carried-thread state
- `3.7` for repost visual metrics

This bug set cuts across those areas, but the failure mode is narrower and more
specific:

- a one-hop repost thread must stay coherent when the original author and the
  repost recipient are not direct contacts
- the original author must still receive repost-thread comments and hearts from
  that non-contact repost recipient
- the repost recipient must still render the original author's real avatar from
  the repost payload instead of falling back to ring/default identity

That deserves a dedicated phase doc so we can lock the exact topology, repair
the trust and fanout contracts, and verify compatibility without widening into
unrelated repost work.

Suggested file:
- `UI-21-POST/3.8-Repost-author-fidelity-and-thread-continuity.md`

## Goal

Fix the exact user-reported topology:

- user A is friends with user B
- user B is friends with user C
- user A is **not** friends with user C

When A creates a post, B receives it, and B reposts it to C:

- C must see A's real profile picture on the repost card
- C's heart on that repost must reach both B and A
- C's comment on that repost must reach both B and A

The desired outcome is:

- repost delivery remains one-hop and privacy-bounded
- the original author learns enough durable repost-thread state to trust and
  apply follow-on events from a non-contact repost recipient
- repost follow-on fanout no longer requires every repost participant to exist
  as a direct contact on every other participant's device
- avatar rendering on passed-along cards no longer depends on the receiver
  already knowing the original author as a contact

## Exact User-Reported Regressions

### Problem 1: Original-author avatar missing on repost recipient

- A creates a text+image post
- B receives it
- B reposts it to C
- C receives the text and media, but not A's profile picture
- expected: C sees A's real profile picture on the repost card

### Problem 2: C heart reaches B but not A

- C hearts the reposted post
- B sees the heart reflected
- A does not
- expected: A receives the heart state and its count reflects all repost-thread
  participants

### Problem 3: C comment reaches B but not A

- C comments on the reposted post
- B sees the comment reflected
- A does not
- expected: A receives the comment and the original thread stays converged

## Current Findings

### Repost-thread recipient resolution is still contact-gated

- `resolvePostEngagementRecipients(...)` reconstructs repost-thread peer IDs
  from:
  - original author
  - persisted repost participants
  - passer fallback
- but it then filters those peer IDs through `ContactRepository.getContact`
- in the exact `Aâ†”B`, `Bâ†”C`, no `Aâ†”C` topology:
  - C knows B as a contact
  - C does **not** know A as a contact
  - A therefore drops out of actual follow-on delivery recipients

### Author-side repost-thread trust is also contact-gated

- incoming `post_comment`, `post_reaction`, and `post_comment_reaction`
  handling currently rejects unknown senders before checking repost-thread
  participant state
- so even if a non-contact repost recipient event reaches the original author,
  the author still rejects it as `unknownSender`

### Current repost payload does not durably teach the author about the new
recipient

- the encrypted `post_pass` payload carries `participant_peer_ids`
- current coverage only locks author + passer in that payload
- the author's notification copy does not yet guarantee that the newly explicit
  repost recipient is durably represented as a trusted repost-thread
  participant on the author's side

### Avatar transport exists, but the sender path is still brittle

- repost payloads can carry `original_author_avatar_base64`
- incoming repost handling can persist a Posts-owned avatar snapshot
- `PostCard` can render `originalAuthorAvatarBytes`
- but sender-side repost avatar inclusion currently depends on sender-local
  disk bytes and silently omits the avatar when that loader returns `null` or
  the raw byte length exceeds the current hard bound
- current coverage proves only the happy path with injected bytes; it does not
  lock the exact non-contact repost-recipient topology with a real sender-local
  disk avatar path

## Locked Decisions

- the exact `Aâ†”B`, `Bâ†”C`, no `Aâ†”C` topology is the primary acceptance topology
  for this plan
- do not solve this by creating synthetic contacts or mutating the contacts
  feature from Posts
- keep repost-thread trust in Posts-owned durable state keyed by `post_id`
- a sender may be accepted for repost-thread follow-ons when either:
  - the sender is a direct contact, or
  - the sender is a trusted repost-thread participant for that `post_id`
- blocked direct contacts remain rejected
- archived direct contacts keep the current notification behavior
- do not widen Posts into generic friends-of-friends messaging
- keep one-hop repost depth, `pickPeople` rejection, author-notification
  privacy, and retry semantics unchanged
- avatar self-renderability remains inside the encrypted repost contract and
  Posts-owned persistence; do not widen into a cross-feature non-contact avatar
  cache
- add explicit FLOW diagnostics around repost avatar load/omit/persist/hydrate
  so device logs identify the failing branch

## Non-Goals

This plan does not cover:

- new repost visual-state or share-count work from `3.7`
- repost sender-speed tuning
- broader contact-system trust changes
- generic friends-of-friends messaging outside repost threads
- redesigning `PostCard`
- widening repost fanout beyond one hop
- changing media ACL rules beyond the smallest avatar-related sender-path
  hardening needed for self-renderability

## Residual Risks To Track

- rollout compatibility for author devices that receive a non-contact repost
  participant event before the richer participant contract arrives
- preserving blocked/archived contact behavior while allowing non-contact
  repost-thread participants
- keeping repost-thread trust scoped strictly by `post_id` so the app does not
  accidentally create a broader friends-of-friends trust model
- avatar payload size and sender-side omission behavior on realistic mobile
  profile pictures
