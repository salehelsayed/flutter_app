# Posts Improvement TDD Plan

Prepared on: 2026-03-15
Prepared by: Codex
Depends on:
- `UI-21-POST/3-Post-improvement.md`
- `UI-21-POST/Plan-Phases.md`

## Goal

Improve Posts send latency and perceived responsiveness without changing the
current Posts product model.

This plan keeps the current architecture constraints:
- Posts remains a Posts feature, not a chat feature
- Posts remains recipient-aware
- Posts does not migrate onto the current group transport
- Posts improvement work must not change current chat send semantics as a side
  effect

## Audit Triage

This revision keeps the meaningful audit findings and drops the parts that were
too absolute for the current codebase.

Accepted:
- extracting a shared chat-first peer-delivery helper before Posts work is
  mis-sequenced and should be deferred
- optimistic send needs an explicit local-create contract instead of silently
  redefining `sendPost(...)`
- `PostCard` needs a delivery-state UI before optimistic dismiss ships
- `PostsWired` must coalesce `postChanges` updates before recipient-level
  background delivery lands
- Phase 7 is too broad and should be split by event family
- media placeholder behavior must be decided in the media phase
- bounded-concurrency delivery must build envelopes lazily per in-flight slot

Accepted with changes:
- split `sendPost(...)`, but do not make `createLocalPost()` return only a
  `PostModel`; the immediate runner needs resolved recipient context too, and
  that contract should live on a public Posts-owned type rather than exposing a
  file-private helper
- background delivery in Phase 3 is in-session async work started from the UI;
  app-scoped lifecycle wiring is required in Phase 6 for reconnect retry, not
  before
- optimistic dismiss should use the existing `ComposePostSheet` submit-then-pop
  flow; `PostsWired` should return from local creation quickly and start
  background delivery after the sheet callback completes
- runner error handling must be explicit before Phase 3 uses
  `unawaited(runner.execute(...))`; unexpected failures should persist terminal
  post state instead of escaping to the async zone
- stale `sending` states should be reconciled when the Posts surface opens, but
  that is planned work, not something the current code already does

Rejected or downgraded:
- `PostTestUser` breakage is not a blocker if `sendPost(...)` remains as a
  synchronous compatibility facade for existing tests and helpers
- the audit treated `main.dart` wiring as mandatory for the first optimistic
  slice; the current code only requires that the runner use long-lived
  dependencies and never capture widget state
- the runner lifecycle does not need a singleton-versus-new-instance decision
  in the plan; the meaningful constraint is one delivery job per `execute(...)`
  with no app-scoped in-flight manager before Phase 6

## Locked Decisions

### Product and architecture

- The compose sheet should dismiss after optimistic local save, not after
  network completion.
- The post should appear in the feed immediately with `deliveryStatus:
  'sending'`.
- Phase 3 background delivery means in-session async work. App-kill and
  reconnect recovery are deferred to Phase 6.
- `sendPost(...)` remains as a synchronous facade for compatibility with
  existing tests and non-optimistic callers.
- Optimistic UI paths should use a new local-create contract plus a
  Posts-owned delivery runner.
- Do not extract a chat-first shared helper before Posts behavior is working.
  If a small reusable transport primitive is needed earlier, it should be
  Posts-shaped and must not change chat semantics.
- Phase 3 must lock the feed-card delivery UX for `sending`, `partial`, and
  `failed` states.
- Recipient fanout should use bounded parallelism, not strict serial execution
  and not unbounded `Future.wait(...)`.
- Retry in this plan applies first to post-create recipient deliveries.
  Follow-on Post events remain fire-and-forget unless a later plan adds
  persisted per-recipient delivery state.

### Status model

Use the existing top-level post statuses first:
- `sending`
- `sent`
- `partial`
- `failed`

Use the existing recipient-delivery statuses first:
- `pending`
- `delivered`
- `inbox`
- `failed`

Do not add new DB fields or migrations unless Phase 6 tests prove the current
persisted contract is insufficient for reconnect retry, especially for nearby
recipient context.

### Concurrency

- Start serial after the architecture split.
- Add bounded parallelism in a later phase.
- Default target: `4` concurrent recipients.
- Make the limit injectable in tests.
- Build per-recipient envelopes lazily as worker slots open.

## Non-Goals

This plan does not cover:
- a new long-lived feed-channel or community-feed architecture
- migrating Posts onto current group topics
- changing approved Posts envelope schemas in the early slices
- changing Group Messages behavior
- introducing OS-level background workers in the first slice
- changing current chat fast-path behavior as part of Posts work

## Success Criteria

The work is successful when all of the following are true:
- tapping `Post` closes the composer after local save
- the new post appears immediately in the feed as `sending`
- post-create recipient deliveries progress asynchronously after dismiss
- top-level post status updates incrementally to `sent`, `partial`, or
  `failed`
- `postChanges` bursts no longer trigger one full-surface reload per recipient
  update
- recipient fanout no longer scales linearly by strict serialization
- failed or unfinished post-create deliveries can be retried after reconnect
- follow-on Post event fanout stops duplicating bespoke serial loops
- no chat entities are persisted by the Posts path

## Proposed Implementation Shape

## Local-create contract

Introduce a Posts-local creation contract that captures the boundary between
foreground save work and async delivery work.

Candidate location:
- `lib/features/posts/application/send_post_use_case.dart`

Expected responsibility:
- validate the post payload
- resolve the recipient set
- persist the local `PostModel` as `sending`
- persist initial `PostRecipientDelivery` rows as `pending`
- return the locally saved post plus the resolved recipient context needed for
  immediate in-session delivery

Suggested shape:
- `sendPost(...)` remains as a synchronous facade built on top of:
  - `createLocalPost(...)`
  - `PostDeliveryRunner.execute(...)`
- add a small result object such as `CreatedLocalPost` instead of returning
  only `PostModel`
- keep the delivery-job payload on a public Posts-owned type; do not leak
  file-private helper types such as `_ResolvedRecipient` across the new
  boundary

`CreatedLocalPost` should carry enough immediate-run context for the first
background slice, for example:
- `PostModel post`
- resolved recipients
- per-recipient nearby distance when applicable

This keeps optimistic UI simple without forcing Phase 3 to solve the full
persisted retry contract up front.

## Posts delivery runner

Create a Posts-owned delivery orchestrator.

Candidate location:
- `lib/features/posts/application/post_delivery_runner.dart`

Expected responsibility:
- accept a `CreatedLocalPost` or equivalent delivery job
- build recipient-specific envelopes lazily
- send to recipients
- update each `PostRecipientDelivery` as results arrive
- recompute the aggregate `PostModel.deliveryStatus`
- avoid any widget references or hidden global lookups
- behave as a dependency-backed executor for one delivery job per
  `execute(...)`; do not add app-scoped in-flight job management before
  Phase 6

Near-term note:
- a tiny Posts-owned `direct -> inbox` transport helper is acceptable if it
  reduces duplication inside Posts
- do not extract chat's full send pipeline first

## UI orchestration

`PostsWired` should stop awaiting full delivery before dismissing the composer.

Target behavior for text-only posts:
1. local post and recipient-delivery rows are saved
2. `ComposePostSheet` dismisses through its existing `onSubmit -> pop` flow
   once local save returns
3. feed shows the post with `sending`
4. `PostsWired` starts `unawaited(runner.execute(...))` after the compose
   callback returns
5. status updates stream into the feed via repository changes

Phase 3 uses in-session async delivery only. If the app is killed or the node
needs reconnect recovery, Phase 6 owns that behavior.

## Delivery UX contract

Phase 3 must lock a minimal aggregate delivery UX on `PostCard`.

Required behavior:
- `sending` is visible inline on the card itself
- `partial` is visible inline on the card itself
- `failed` is visible inline on the card itself
- the UI only shows aggregate post state, not per-recipient progress
- follow-on post actions are disabled while `deliveryStatus == 'sending'`
- failure is visible in the feed; do not rely on snackbars or toasts as the
  primary signal

Minimal design lock:
- a small footer label or chip: `Sending...`, `Partially sent`, `Send failed`
- optional subtle opacity reduction while `sending`
- no per-recipient progress UI in this plan

## Surface refresh strategy

The current `postChanges -> unawaited(_loadSurface())` path is too expensive
for recipient-level background delivery.

Phase 3 must add a coalesced refresh path in `PostsWired`:
- debounce rapid `postChanges` bursts
- avoid overlapping `_loadSurface()` executions
- allow a trailing refresh after an in-flight load completes

This plan does not require entity-level incremental UI updates yet, but it does
require that recipient updates stop forcing one full feed reload each.

## DI and lifecycle wiring strategy

Use the existing app wiring precedent where appropriate:
- `PendingMessageRetrier` in `lib/main.dart` is the model for app-scoped retry
  services

Implementation guidance:
- Phase 2 and Phase 3 may inject or construct `PostDeliveryRunner` from
  long-lived repositories and services already owned by the app
- the runner must not capture widget state, `BuildContext`, or screen-local
  subscriptions
- Phase 6 wires reconnect retry through `lib/main.dart` using the same
  lifecycle pattern as `PendingMessageRetrier`

## Retry contract note

The current schema is probably sufficient for in-session delivery and may be
enough for reconnect retry, but that must be proven in Phase 6.

Phase 6 tests must explicitly verify that retry can rebuild delivery work for:
- `allFriends`
- `pickPeople`
- `peopleNearby`

If nearby retry needs one extra persisted field or query, Phase 6 may add the
smallest missing contract then. Do not assume this up front and do not widen
earlier phases for it.

## TDD Strategy

Use small RED/GREEN slices. Do not attempt the whole refactor in one pass.

Test layers to use:
- unit tests for local-create logic
- unit tests for the Posts delivery runner
- widget tests for composer dismissal and feed state transitions
- integration-style tests with fake P2P harnesses for concurrency and retry

Existing harnesses already available:
- `test/shared/fakes/fake_p2p_network.dart`
- `test/shared/fakes/fake_p2p_service_integration.dart`
- `test/core/services/fake_p2p_service.dart`
- existing Posts repository and use-case tests under `test/features/posts`

Compatibility rule:
- keep `sendPost(...)` usable by existing synchronous tests until optimistic UI
  has dedicated coverage

## Revised Phase Sequence

1. Phase 1: split local creation from delivery execution while keeping
   `sendPost(...)` as a synchronous facade
2. Phase 2: introduce `PostDeliveryRunner` with serial in-session behavior
3. Phase 3: optimistic dismiss for text-only posts, delivery-state UI, and
   coalesced surface refresh
4. Phase 4: bounded parallel fanout for text-only posts
5. Phase 5: optimistic media posts with explicit upload-placeholder behavior
6. Phase 6: reconnect retry and lifecycle-scoped recovery
7. Phase 7a-d: migrate follow-on event fanout in narrower slices
8. Phase 8: optional shared helper extraction only if Posts and chat paths
   actually converge

## Phased Plan

## Phase 1: Split local creation from delivery execution

Outcome:
- Posts send has an explicit foreground-save boundary
- existing synchronous callers still work through `sendPost(...)`

### RED

Suggested test files:
- `test/features/posts/improvement/create_local_post_use_case_test.dart`
- `test/features/posts/improvement/send_post_facade_test.dart`

Minimum RED cases:
- local creation persists `PostModel` with `deliveryStatus: 'sending'`
- local creation persists `PostRecipientDelivery` rows as `pending`
- local creation returns the resolved recipient context needed by the immediate
  runner
- `sendPost(...)` still returns terminal `SendPostResult` values when it is
  used as the synchronous facade
- no `ConversationMessage` is written during Posts send

### GREEN

Split current `sendPost(...)` into:
- local optimistic creation logic
- delivery execution logic
- a synchronous facade that composes both for compatibility

Possible shape:
- `createLocalPost(...) -> CreatedLocalPost`
- `PostDeliveryRunner.execute(CreatedLocalPost job)`
- `sendPost(...)` calls both and awaits runner completion

### REFACTOR

- move aggregate-status calculation into one helper
- remove inline delivery-boundary ambiguity from `send_post_use_case.dart`

### Exit Gate

- synchronous send behavior still matches current tests
- optimistic UI now has a safe creation boundary to target

## Phase 2: Introduce `PostDeliveryRunner` with serial in-session behavior

Outcome:
- delivery execution is separable from local creation
- background delivery can run without widget-owned logic

Do not parallelize yet in this phase.

### RED

Suggested test file:
- `test/features/posts/improvement/post_delivery_runner_test.dart`

Minimum RED cases:
- runner updates recipient rows as network results arrive
- runner recomputes top-level post status after each recipient update, not only
  at the very end
- all-success, mixed, and all-failed aggregates are correct
- unexpected runner exceptions are caught and persisted as a terminal post
  failure instead of escaping the async boundary
- runner does not touch widget state
- runner does not write chat entities

### GREEN

Implement `PostDeliveryRunner`.

Implementation notes:
- keep serial behavior first
- use only long-lived dependencies such as `PostRepository`, `ContactRepository`,
  `P2PService`, and `Bridge`
- a small Posts-owned `direct -> inbox` helper is acceptable if it keeps logic
  out of the runner
- the runner must own its error handling and finalize post state on internal
  failures because Phase 3 starts it with `unawaited(...)`
- do not rewrite chat to use this code in this phase

### REFACTOR

- isolate envelope building from transport attempts
- keep aggregate status calculation in one place

### Exit Gate

- the delivery runner can replace the inline loop in `send_post_use_case.dart`
- optimistic UI has a background-safe in-session executor

## Phase 3: Make text-only Posts dismiss optimistically

Outcome:
- text-only Posts no longer block the composer
- the feed can reflect send state without thrashing

This is the first user-visible improvement slice.

### RED

Suggested test files:

- extend `test/features/posts/phase1/compose_post_sheet_test.dart`
- extend `test/features/posts/phase1/posts_wired_test.dart`

Minimum RED cases:
- composer closes after local save, before network completion
- new text post appears in feed as `sending`
- post card shows inline `sending`, `partial`, and `failed` states
- follow-on card actions are disabled while a post is still `sending`
- background runner continues after composer dismisses in the current session
- feed updates to `sent`, `partial`, or `failed` after runner completion
- repository bursts do not trigger one full `_loadSurface()` per recipient

- opening the Posts surface reconciles lingering `sending` posts against
  current recipient-delivery rows before rendering stale state forever

### GREEN

Update:
- `lib/features/posts/presentation/screens/posts_wired.dart`
- `lib/features/posts/presentation/widgets/post_card.dart`

Implementation notes:
- use `createLocalPost(...)` on the foreground path
- keep the existing compose-sheet dismiss contract: `onSubmit(...)` should only
  await local creation, then let `ComposePostSheet` pop normally
- start `unawaited(runner.execute(...))` from `PostsWired` after the submit
  callback returns
- add a debounced and serialized `_loadSurface()` path
- keep this phase limited to in-session async delivery

### REFACTOR

- isolate background-start logic behind one `PostsWired` method
- keep delivery-status presentation logic out of orchestration code

### Exit Gate

- text-only Posts no longer feel blocked by network send
- send state is visible in the feed
- full-surface reload churn is controlled well enough for recipient-level

## Phase 4: Add bounded parallel fanout for text-only Posts

Outcome:
- recipient loop no longer scales linearly by strict serialization

### RED

Suggested test file:
- `test/features/posts/improvement/post_delivery_runner_parallel_test.dart`

Minimum RED cases:
- runner never exceeds the configured concurrency limit
- recipient completions update rows incrementally
- aggregate top-level status is correct for:
  - all delivered
  - mixed delivered and failed
  - all failed
- recipient-specific encryption still produces one envelope per recipient where
  required
- envelope building happens lazily as worker slots open

### GREEN

Implement bounded concurrency inside `PostDeliveryRunner`.

Do not use:
- strict serial loop
- unbounded `Future.wait(...)`

Implementation notes:
- make concurrency limit constructor-injected
- default to a constant in Posts application code
- use a simple pool pattern with at most `N` in-flight futures
- do not add a new dependency package for the first pass unless tests show the
  inline pool is too error-prone

### REFACTOR

- extract a small worker-pool helper if needed
- keep business-state aggregation separate from scheduling

### Exit Gate

- total send latency improves for multi-recipient text posts
- tests prove the concurrency cap

## Phase 5: Extend optimistic background delivery to media posts

Outcome:
- media posts follow the same perceived-send model as text posts

This phase is intentionally after text-only because media adds preparation and
upload complexity.

### RED

Suggested test files:
- `test/features/posts/improvement/send_post_media_background_test.dart`
- extend `test/features/posts/phase2/attach_post_media_use_case_test.dart`
- extend `test/features/posts/phase2/compose_post_sheet_media_test.dart`

Minimum RED cases:
- composer dismisses after local post creation even when media exists
- local post can exist in a media-uploading skeleton state before remote blob
  ids are available
- media-only skeleton posts are not visually blank in the feed
- media preparation and upload continue after dismiss
- upload failure marks post `failed` without losing the local post
- successful upload allows recipient fanout to proceed
- multi-image posts preserve media order

### GREEN

Update:
- `lib/features/posts/application/attach_post_media_use_case.dart`
- Posts delivery orchestration to allow media preparation before recipient
  envelopes are sent but after local optimistic save
- `lib/features/posts/presentation/widgets/post_card.dart`

Design lock for this phase:
- a media-uploading skeleton must render an explicit placeholder treatment
- a media-only optimistic post must not look like an empty card
- the card should show an inline upload-state label until media metadata is
  ready

Implementation note:
- keep media upload once per post
- keep post-specific media allowlists intact
- do not fake a conversation-media flow
- treat media posts as a two-stage state machine:
  1. local skeleton post is saved immediately
  2. media metadata is attached after upload succeeds
  3. recipient fanout starts after uploaded metadata exists

### REFACTOR

- isolate media-preparation stage from recipient-send stage
- ensure media and recipient delivery errors are surfaced separately

### Exit Gate

- media posts use the same perceived-send UX as text posts
- media-only optimistic posts stay understandable in the feed

### REFACTOR

- share reconnect-trigger patterns with chat retrier where appropriate

### Exit Gate

## Guardrails

- Do not change Posts envelope schemas in the early slices.
- Do not change feed ordering rules as part of this effort.
- Do not introduce chat repository writes into Posts.
- Do not use the current group publish path for Posts.
- Do not use unbounded fanout concurrency.
- Do not hide delivery failures by dropping local posts.
- Do not rewrite chat to use a shared helper in Phases 1-7.
- If a shared helper is extracted later, it must preserve current chat
  fast-path behavior.

## Open Technical Risks

### 1. Retry contract for nearby posts

Nearby sends currently use per-recipient distance at envelope-build time.
Phase 6 must prove the persisted contract is sufficient for reconnect retry or
add the smallest missing piece.

### 2. Media-phase complexity

Media posts require careful sequencing:
- local post must appear immediately
- media upload must still happen before recipient envelopes are fully
  deliverable

### 3. Bridge pressure

Bounded concurrency is intentionally conservative because the bridge does not
advertise batching or explicit backpressure behavior.

### 4. Feed churn

Incremental recipient updates currently map to full feed reloads. Phase 3 must
coalesce that path before Phase 4 increases recipient overlap.

## Definition Of Done

This plan is complete when:
- `sendPost(...)` has an explicit local-create boundary
- text-only Posts dismiss optimistically
- recipient delivery runs in bounded parallel background work
- media posts follow the same perceived-send UX
- retry exists for unfinished post-create delivery
- follow-on Post events use one Posts delivery model instead of bespoke loops
- chat semantics remain unchanged unless an optional later helper extraction is
  explicitly proven safe
