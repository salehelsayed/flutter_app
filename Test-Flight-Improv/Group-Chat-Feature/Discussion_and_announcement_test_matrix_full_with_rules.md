# Libp2p Discussion and Announcement Group Test Matrix with Coverage Rules

## Scope

This file narrows the broader group-chat test inventory down to the exact user-visible discussion and announcement behaviors that must be proven before this feature is trustworthy in the current Flutter product.

It is meant to sit beside, not replace:

- `Test-Flight-Improv/Group-Chat-Feature/Discussion-And-Announcement-Feature-Audit.md`
- `Test-Flight-Improv/libp2p_group_chat_test_matrix_full_with_rules.md`
- `Test-Flight-Improv/Message-Bubble-List-1_1-Feature/message_context_overlay_test_matrix_full_with_rules.md`
- `Test-Flight-Improv/Group-Chat-Feature/C4-01-Create-Discussion.md`
- `Test-Flight-Improv/Group-Chat-Feature/C4-02-Send-Message.md`
- `Test-Flight-Improv/Group-Chat-Feature/C4-03-Receive-Message.md`
- `Test-Flight-Improv/Group-Chat-Feature/C4-04-Invite-And-Join.md`
- `Test-Flight-Improv/Group-Chat-Feature/C4-05-Recovery-And-Reliability.md`

The goal is not only delivery correctness. The goal is user trust: membership changes should be understandable, identities should be human-readable, announcement permissions should stay explicit, long-press message actions should feel coherent, mixed-friendship groups should still work, and offline replay should be private, membership-aware, seamless for text/image/video/voice, and reliable under recovery.

## Actors

- **A** = group creator or current admin
- **B** = current member or announcement reader
- **C** = invitee, removed member, or non-friend current member
- **D** = prospective member not already in the group
- **X** = unauthorized peer, unknown inviter, or relay/storage operator

## Priority guide

- **P0** = release-blocking; core membership trust, permissions, identity, delivery, and offline privacy
- **P1** = important; should be covered before broad rollout or before claiming the UX is complete
- **P2** = optional or product-dependent; cover if the feature is intentionally in scope

## Minimum discussion and announcement rules and invariants

These are the baseline rules this matrix assumes:

- Discussion groups allow current members to write.
- Announcement groups allow only admins to write.
- Membership changes must converge and remain visible enough for humans to understand who was added, who joined, who was removed, and who became admin.
- Creator/admin identity should resolve to a human-readable username when the username exists.
- Supported group messages should expose one coherent long-press context surface instead of splitting core actions across unrelated gestures without clear product intent.
- Group transport differences do not by themselves block local-only long-press affordances such as selected-message preview, reply entry, or copy.
- If group per-message edit/delete remain unsupported, the group long-press surface should still expose the supported actions honestly rather than disappearing entirely.
- Group send/receive is membership-based once membership exists; it should not require direct friendship at message time.
- Media fan-out follows group membership state, not contact state.
- Offline replay, retry, and reconnect behavior must converge without duplicate visible messages or ghost membership.
- Relay-backed offline replay should not expose plaintext message or membership-event content to relay operators.
- Encrypted offline replay must keep current-version group messaging seamless for text, quote replies, image, video, GIF/file attachments, and recorded voice, rather than protecting only plain text.
- Members added later should only decrypt the backlog allowed by the relevant membership or key epoch, and members who leave or are removed should not decrypt replay for newer traffic.
- Legacy plaintext replay compatibility for old app versions is intentionally out of scope for this matrix revision.
- Current product constraints may still include contact-only pickers and unknown-sender invite rejection unless a later product decision expands non-friend onboarding.
- Optional features, if supported, should behave explicitly rather than implicitly: admin transfer, search, pinning, per-message edit/delete, read receipts, moderation, scheduled announcements, and analytics.

## Current repo execution note (2026-04-10)

Use this note when turning the matrix into repo-local work:

- Strong current repo-owned coverage already exists for discussion and announcement create flows, admin-only announcement write control, send reliability, media delivery, invite review accept/decline/expiry, member removal visibility, promotion/demotion, per-group mute, dissolve/read-only history, and offline catch-up/recovery.
- Direct 1:1 conversations already ship the shared `MessageContextOverlay` with selected-message preview, reactions, reply, copy, edit, delete, and deep widget/screen coverage. Group conversations do not adopt that overlay today; they only open `ReactionBar` on long-press, while quote reply is handled separately by swipe on incoming rows.
- The main current repo-local trust gaps are: the shipped batch `members_added` path does not create a durable in-chat add-members event, pending-invite acceptance does not create a durable `X joined` or `X accepted` event, creator/admin usernames can still fall back to peer IDs, group long-press message actions lag far behind the 1:1 surface, non-friend onboarding and avatar resolution are incomplete, group create still has a defensive branch that can save a locally keyless group if bridge key return and fallback keygen both fail, zero-peer plus inbox-fail sends rely on full failed-message resend rather than the inbox-only retry lane despite persisting `inboxRetryPayload`, invite-accept replay currently omits offline reactions from the immediate catch-up path, invite-accept bridge timeouts can leave the group in an accepted-but-not-yet-live state without direct convergence proof or explicit retry UX, and group relay inbox replay payloads are still plaintext rather than encrypted, membership-aware, and media-complete.
- Several higher-level product capabilities are not currently landed in this repo: explicit admin transfer, group search, message pinning, per-message edit/delete/tombstone, read receipts, member moderation mute/ban, scheduled announcements, announcement edit-after-send/delete-after-send, and analytics.
- Execution rule for this matrix: use it to judge trust at the discussion/announcement product surface. Do not reopen already-closed generic transport rows unless they directly affect these user-visible flows. For long-press parity, separate UI-host gaps from true transport/protocol dependencies instead of treating every missing group action as transport-bound.

## Coverage policy used in this matrix

### Coverage legend

- **Required** = should exist for this journey before you treat the feature as production-ready.
- **Recommended** = high-value coverage, but not mandatory for every release gate.
- **N/A** = do not force this layer for this journey.

### Rules

**Unit**  
Use for logic-heavy pieces:

- identity fallback rules
- role and write-permission checks
- membership-event dedupe and replay protection
- non-friend eligibility and picker filtering rules
- relay inbox payload construction and privacy rules
- notification suppression or visibility rules tied to group state

**Integration**  
Use for most journeys:

- create discussion or announcement groups
- add, invite, accept, decline, remove, and promote flows
- member list and conversation identity rendering
- discussion send/receive and announcement read-only enforcement
- media fan-out, offline replay, and recovery behavior

**Smoke**  
Keep this small and release-blocking:

- create discussion group
- create announcement group
- discussion online fan-out
- announcement admin-only compose
- add member and show visible group-state change
- remove member and show visible group-state change
- offline/retry recovery for one representative group message path

**Fake Network**  
Use where network behavior is the main risk:

- partial fan-out
- offline invite or offline member replay
- reconnect and cursor-based backlog drain
- duplicate or stale membership-event delivery
- post-removal cut-off
- relay/store-and-forward privacy or retention boundaries

**3-Party E2E (3 simulators)**  
Use for user-visible A/B/C flows:

- A creates a group with B and C
- A adds or removes C while B observes
- A runs an announcement group where B reads and A writes
- a non-friend member path or mixed-social-graph path if the product supports it
- member list, admin badges, and timeline events converge across devices

### Matrix interpretation

- **Integration** is Required for most rows because discussion and announcement trust is emergent across UI, local persistence, membership state, transport, and recovery.
- Rows that are strongly UI-facing should also have widget coverage in practice, even though widget tests are not a dedicated column in this matrix.
- Some privacy rows need bridge or Go-layer proof in addition to Flutter tests because the user-visible trust claim depends on payload shape, not only the Dart UI.
- Optional P2 rows still assume the feature exists. If the capability is not shipped in this product, mark the row **Unsupported** instead of forcing immediate test work.

## Missing journeys added in this revision

Compared with the broader group-chat matrix, this revision explicitly sharpens the discussion and announcement journeys that most affect user trust:

- durable `members added` timeline visibility for the shipped batch add path
- durable `invite accepted` or `member joined` timeline visibility after pending-invite acceptance
- durable `member left the group` timeline visibility for voluntary leave, separate from admin removal
- creator/admin username backfill instead of peer-ID fallback
- long-press context overlay parity with the mature 1:1 `MessageContextOverlay`
- long-press reply/copy support for groups even before group edit/delete ships
- graceful fallback when reactions are unavailable instead of disabling long-press entirely
- single stable group-message bubble rendering without a doubled or stacked shell artifact
- if inline Feed group-thread actions remain in scope, they need their own reaction and permission parity contract instead of being treated only as navigation into the full conversation
- cross-friendship participation and avatar visibility expectations
- bootstrap key integrity so group create never succeeds into a locally keyless state
- explicit recovery ownership for `failed + inboxRetryPayload` rows created by the zero-peer plus inbox-fail branch
- immediate reaction convergence for offline replay triggered by pending-invite acceptance
- invite-accept bridge-timeout recovery that proves locally persisted accepted groups later become live joined topics without needing the invite row again
- relay inbox encryption for offline group replay payloads, including membership-bound add/remove/leave behavior and seamless text/media/voice catch-up
- current product-scope clarity for search, pinning, read receipts, moderation, and scheduled announcement tooling

## Notes for implementation

- For every membership-visibility row, assert both the structural truth and the human-visible truth: member list, role badges, and durable conversation history should all agree.
- For voluntary-leave rows, do not collapse explicit leave broadcast into generic cleanup by default. `leaveGroup()` is also reused when a remote `member_removed` event forces local cleanup, so naive broadcasting at the cleanup layer risks duplicate or false `left the group` events.
- For long-press parity rows, do not blame group transport by default. Selected-message preview, reply entry, copy, and action ordering are mostly UI-host concerns; reserve transport/protocol scope for any future group edit/delete behavior.
- For reaction-transparency rows, prove that inspection and mutation are separate user-visible contracts. In a multi-user group, tapping a reaction cluster to learn who reacted should not double as a silent remove-my-reaction gesture.
- For cross-entry-point rows, do not assume one fix in `GroupConversationWired` is enough without proof. The same message-level behavior must be exercised from both `Orbit` and `Feed`, because both surfaces open the group conversation in real product use.
- For inline Feed rows, test the actual `FeedCard` surface directly. Navigation from Feed into `GroupConversationWired` does not prove the card-level reaction bar, chip taps, or read-only gating behave correctly.
- For mixed-social-graph rows, prove both transport behavior and surface behavior; it is not enough for messages to arrive if identity or avatar rendering still looks broken or anonymous.
- For visual rendering rows, prove the actual row shell, not only message content. A widget can still show the right text while rendering an obviously wrong doubled background.
- For lifecycle-send rows, prove what happens after the initiating route is gone. Generic transport reliability is not enough if the UI-level bg-task or completion-order contract can still strand the final row in the wrong state.
- For retention-truth rows, prove both the storage policy and the user-facing truth. Expired non-system backlog must stay gone, retained backlog must stay visible, old system membership events must still converge, and the banner/list copy must match what actually happened.
- For bootstrap-key rows, do not assume the current Go bridge contract is enough forever. Prove that create either persists usable local key state or fails honestly, and that invite/send/rejoin flows cannot continue from a silently keyless local group.
- For retry-owner rows, prove which job owns the branch. If a row carries `inboxRetryPayload` but is marked `failed`, either full resend must explicitly recover it or the inbox-only retry query must own it; do not leave recovery semantics implicit across two retry systems.
- For invite-accept replay rows, prove the first user-visible catch-up window, not only eventual consistency after startup or resume. If reactions are intentionally deferred until a later drain, document that contract explicitly and prove the later sweep closes the gap.
- For invite-accept bridge-timeout rows, do not overclaim permanent orphaning. Prove the narrower contract: the accepted group may be locally present before live join finishes, automatic startup/resume/watchdog rejoin must close that gap promptly, and the UI must stay honest about degraded recovery or expose an explicit retry path if convergence stalls.
- For offline privacy rows, inspect the actual stored replay payload or bridge request shape, not only the UI outcome.
- For encrypted replay rows, prove both privacy and continuity: the same solution must stay correct across add/remove/leave boundaries and across text, reply, image, video, and recorded voice recovery.
- Do not weaken the encrypted replay contract for old-build convenience in this matrix. Judge trust on the supported current release family, not on legacy plaintext compatibility.
- For rows that are currently unsupported, keep them explicitly unsupported in tracking until product scope changes; do not silently convert them into implied bugs.
- For 3-party E2E work, collapse roles where useful: A = admin, B = existing member, C = invitee or removed member.
