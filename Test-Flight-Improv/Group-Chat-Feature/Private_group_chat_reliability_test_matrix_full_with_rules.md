# Private Group Chat Reliability Test Matrix with Coverage Rules

## Scope

This file is a private-group-chat reliability matrix derived from the uploaded bridge, event-dispatch, media, and Dart bridge-helper files, using the formatting style of `Discussion_and_announcement_test_matrix_full_with_rules_COMPLETE.md`.

The immediate failure pattern to guard against is: members are added, removed, and re-added, and later one or more active members stop seeing incoming group messages. The tests below treat that as a cross-layer reliability problem, not only as a transport problem.

The goal is end-to-end user trust:

- active members must receive every message they are entitled to see
- removed members must not receive or decrypt newer content
- re-added members must move to the current config and key epoch before being treated as active
- stale config, stale key, old invites, and out-of-order events must not downgrade working peers
- live pubsub, offline inbox, replay, history repair, media, reactions, UI state, notifications, and diagnostics must agree
- no bridge timeout, EventChannel failure, dispatcher overflow, or relay outage may become silent message loss

## Actors

- **A** = group creator or current admin
- **B** = stable active member and observer
- **C** = member being removed and re-added
- **D** = newly invited member or additional churn target
- **X** = unauthorized peer, stale invite holder, relay/storage operator, or attacker

## Priority guide

- **P0** = release-blocking reliability, privacy, or diagnosability contract
- **P1** = important before broad rollout or before claiming the feature is complete
- **P2** = product-dependent or lower-risk polish

## Minimum private group rules and invariants

- A peer is an active member only when Flutter state, Go group config, topic join state, and current group key epoch agree or the app shows an explicit pending/degraded state.
- Group key epochs are monotonic. A stale lower epoch must never overwrite a newer epoch.
- Same-epoch different-key conflicts must be rejected or repaired; they must not silently split the group.
- Re-add is not a normal add. It must close the removed interval, install the latest config/key, and reject delayed old remove/key/config events.
- The bridge must not return a success that leaves Go validators on stale key/config.
- Live delivery and replay must dedupe into one visible row by stable message id.
- Offline inbox replay must be membership-bound by interval and key epoch.
- Removed members may retain allowed local old history but must not receive, decrypt, react to, or be notified about post-removal content.
- Dispatcher overflow and EventChannel failure are reliability events that require recovery. They are not acceptable silent-drop conditions.
- Diagnostics must identify whether a miss came from transport, topic join, stale config, stale key, validator rejection, decryption failure, replay, DB persistence, or UI filtering.
- Secrets such as group keys, private keys, plaintext, and decrypted invite content must never appear in logs, diagnostics, relay payloads, or media metadata.

## Code-level reliability risks observed in the uploaded files

These observations determine the highest-priority tests:

1. `bridge.go:1539-1544` returns `{ ok: true, note: "ALREADY_JOINED" }` when `JoinGroupTopic` says the topic is already joined, but the bridge does not apply the newer `groupConfig`, `groupKey`, or `keyEpoch` from the join payload. This can strand a re-added or recovered member on stale state.
2. `bridge.go:1934-1937` applies `GroupUpdateKey` directly with the supplied epoch. The uploaded bridge shows no monotonic guard against stale lower epochs or same-epoch different-key conflicts.
3. `bridge.go:1791` applies `GroupUpdateConfig` directly. The uploaded bridge shows no version, authority, or stale-config guard at the bridge boundary.
4. `bridge.go:1889-1893` computes `GroupGenerateNextKey` from Go in-memory key state and defaults to epoch 2 if missing. After restart, this can fork epochs unless Flutter restores the latest key before generation.
5. `bridge_group_helpers.dart:80-98` contains a legacy `callGroupJoin` that sends only `topicName`, while `bridge.go:1502-1531` requires `groupConfig`, `groupKey`, and `keyEpoch`.
6. `bridge.go:356-361` ignores later `Initialize` callbacks when the singleton node already exists. If native callback identity can change, group events can be routed to stale callback state.
7. `event_dispatcher.go:109-117` drops non-coalesced events when the message queue is full. That must trigger replay recovery or active members can miss incoming messages.
8. `callGroupKeygen` and `callGroupDecrypt` read success fields without first checking `ok`; bridge failures can become unclassified Dart errors unless callers wrap them.
9. Group inbox retrieve responses expose only outer `from`, `message`, and `timestamp`; dedupe and membership interval checks must be enforced after decrypting the envelope.
10. Media upload uses `AllowedPeers`; tests must prove that list is current active membership, especially after remove and re-add.

## Coverage policy used in this matrix

### Coverage legend

- **Required** = should exist before private group chat is treated as production-ready
- **Recommended** = high-value additional proof
- **N/A** = not required for this row

### Rules

**Unit**
Use for deterministic logic-heavy parts: epoch monotonicity, stale config rejection, invite validity, message-id dedupe, membership interval checks, bridge payload shape, secret redaction, and helper error handling.

**Integration**
Use for most rows: create, join, add, remove, re-add, publish, key rotation, inbox store/retrieve, local persistence, Flutter bridge routing, and UI state.

**Smoke**
Keep small and release-blocking. Smoke proves the core story: create, add, remove, re-add, send, replay, and no removed-member leakage.

**Fake Network**
Use wherever network order is the risk: partitions, delayed key/config, duplicate live plus replay, zero topic peers, dispatcher overflow, stale invite delivery, relay failures, and cursor replay.

**3-Party E2E**
Use for user-visible A/B/C flows: A creates, B observes, C is removed/re-added, all members send and receive, and the timeline/member list/notifications converge.

## Failure-isolation runbook for “member cannot see incoming messages”

When a test or manual session reproduces the bug, capture and classify it in this order:

1. **Callback path**: verify EventChannel is alive, `onGroupMessageReceived` is still set, and native `Initialize` did not retain a stale callback.
2. **Topic path**: verify the affected peer is joined to the group topic after restart/reconnect/re-add.
3. **Config path**: compare Flutter active member list with Go group config. Look especially for `ALREADY_JOINED` joins that did not refresh config.
4. **Key path**: compare local Flutter key epoch and Go validator key epoch. Check for stale lower epoch updates or same-epoch different-key conflicts.
5. **Validator path**: inspect `group:validation_rejected`, `group:decryption_failed`, and `group:payload_parse_failed` diagnostics.
6. **Replay path**: verify group inbox store recipients, cursor state, history gaps, and dedupe by message id.
7. **Persistence path**: verify the message was not received but then filtered, failed to persist, or hidden by membership interval logic.
8. **UI path**: verify route/widget lifecycle, unread state, mute state, and removed/pending member filters.

## Notes for implementation

- Build the P0 rows with a deterministic fake network first. E2E simulator tests are essential, but fake-network ordering control is what will expose stale config/key races.
- Add an invariant checker to every churn test: for each message, compute the expected recipient set from membership intervals and assert delivered visible rows match exactly.
- Treat `ALREADY_JOINED` as unsafe until a test proves the latest config and key are also installed. A success response from join must mean the node can decrypt and validate current messages.
- Gate key updates with monotonic epoch rules. Lower epoch is stale, same epoch same key is idempotent, same epoch different key is conflict, higher epoch requires valid authority and membership context.
- Add operation ids or config versions for membership changes so old remove/config events cannot revoke a newer re-add.
- Never advance replay cursor or ack state until decrypted message persistence succeeds.
- Do not treat pubsub publish success or `topicPeers > 0` as recipient delivery. It is only one branch of delivery truth.
- Dispatcher overflow must be connected to replay recovery. A diagnostic without recovery is still message loss.
- Use sentinel secrets in tests and scan logs so added diagnostics do not leak group keys or plaintext.
- Every row involving removed or re-added members should assert both structural truth and visible truth: member list, timeline event, compose permission, unread count, notifications, and message rows.
