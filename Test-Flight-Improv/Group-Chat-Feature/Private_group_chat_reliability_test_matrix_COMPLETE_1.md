# Private Group Chat Reliability Test Matrix with Coverage Rules

## Scope

This matrix is a release-gate test plan for the private group chat feature in this repo. It is intentionally focused on the failure class you described: members being added, removed, and re-added, after which some current members stop seeing incoming messages. The matrix covers the Go transport/crypto/inbox/relay code and the Flutter app-layer integration tests around it.

The matrix cannot prove every possible internet failure away, but it is designed so that passing the P0 rows gives strong evidence that current members reliably receive entitled messages, removed members lose access, re-added members regain access with the right epoch, live PubSub loss is repaired by inbox replay, and all membership/key/device races converge deterministically.

## Actors

- **A** = group creator/admin or current sender.
- **B** = stable current member who must continue receiving messages.
- **C** = member being removed, offline, re-added, or using a stale key/config.
- **D** = late invitee/new member.
- **S1/S2** = sibling devices for the same logical user.
- **X** = unauthorized peer, removed member after cutoff, revoked device, malicious relay/source, or malformed sender.

## Status legend

- **Covered** = a direct automated test already exists in your full repo and proves the user-visible contract.
- **Partial** = adjacent low-level coverage exists, but the full transport + membership + inbox + app outcome is not proven.
- **Open** = add this test or explicitly map it to an equivalent existing test before treating private group chat as reliable.
- **Unsupported** = capability is intentionally not supported and should be blocked or documented.

## Priority guide

- **P0** = release-blocking reliability, privacy, or authorization contract.
- **P1** = important hardening before broad rollout or before claiming reliability.
- **P2** = useful or product-dependent coverage.

## Minimum private group chat invariants

- A current member must eventually receive every message they are entitled to, either live through PubSub or later through inbox replay.
- A removed member must not send accepted post-removal messages and must not decrypt post-removal replay.
- A re-added member must regain access only from the correct rejoin membership/key epoch window.
- Add/remove/re-add must update membership, key epoch, recipient list, PubSub validator, discovery targets, topic rejoin state, and replay cutoffs together.
- Live PubSub delivery and durable inbox replay must dedupe into one visible message, not two.
- Validator behavior for live messages and replay behavior for stored messages must enforce the same sender, device, role, membership, and key rules.
- Multi-device sender identity must be bound to the active device transport peer, not only to a logical member ID.
- Stale local config/key state must produce a repairable diagnostic, not silent message loss.
- Relay refresh or watchdog restart must either preserve joined topics or force explicit app rejoin before clearing recovery state.
- Test artifacts must make every missing-message failure class attributable to one of: config mismatch, key epoch mismatch, device binding, topic peer absence, inbox failure, replay dedupe/cutoff, or relay recovery.

## Code-review risk anchors that shaped this matrix

- `hasKeyRotationGrace` requires `PrevKeyEpoch > 0` (`pubsub.go:414-419`). If initial groups use epoch `0`, the first rotation from epoch `0` to `1` may reject in-flight epoch-0 messages during grace. This is a high-priority suspected reliability bug.
- `UpdateGroupConfig` stores any pointer including `nil` (`pubsub.go:356-360`), while `findMember` dereferences `config.Members`. Nil or malformed config tests are required to prevent panics/black holes.
- `JoinGroupTopic` registers a validator before join/subscribe (`pubsub.go:70-84`). Failure paths need tests proving validators are not leaked and later rejoin works.
- Device-aware validation exists (`activeMemberDeviceForEnvelope`), but known-member dialing and expected-peer counting use `member.PeerId` (`pubsub.go:1202-1242`, `1437-1545`, `1675-1722`). Multi-device or re-added-device cases need direct tests.
- The subscription handler skips messages where `env.SenderId == selfPeerId` (`pubsub.go:702-718`). This must be tested with sibling devices and any account-level sender ID model.
- Signature data covers only `groupId|epoch|ciphertext` (`group.go:107-111`). Tamper tests must define whether type, nonce, sender, and device-field mutations are rejected by validator/decryption or are harmless.
- `GroupInboxStore` trusts caller-provided `recipientPeerIds` and ignores push title/body (`group_inbox.go:136-222`). App-layer tests must enforce current-membership recipients and relay privacy.
- `GroupHistoryRepairRange` passes integrity values through to the app layer (`group_inbox.go:277-323`). App tests must reject range/head mismatches.
- `Stop` clears group runtime state (`node.go:430-463`). Relay watchdog/full restart must be paired with explicit app rejoin coverage.

## Coverage policy used in this matrix

### Coverage legend

- **Required** = should exist before this feature is considered production-ready.
- **Recommended** = high-value additional proof.
- **N/A** = not necessary for that row.

### Rules

**Unit**: use for crypto, signature, envelope parsing, key epoch/grace, authorization helpers, device binding, discovery filters, retry/backoff, request-shape, and relay session state.

**Integration**: use for Node-level publish/receive, join/leave/update, inbox store/retrieve/repair, relay recovery, topic rejoin, app persistence, membership ordering, and replay cutoff behavior.

**Smoke**: keep small but release-blocking: create/join, remove, re-add, send after re-add, offline replay, zero-peer fallback, and relay restart rejoin.

**Fake Network**: use deterministic fake clocks, fake PubSub topic peers, fake relay/inbox/rendezvous, partitions, delayed delivery, duplicated delivery, stale envelopes, and relay restart.

**3-Party E2E**: use A/B/C simulators for every path where the user-visible failure is `some member stopped seeing messages`.
