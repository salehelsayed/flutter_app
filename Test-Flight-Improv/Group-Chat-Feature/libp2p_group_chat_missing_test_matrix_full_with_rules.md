# libp2p Group Chat Missing Test Matrix with Coverage Rules

## Scope

This file is a gap-focused addendum to the current group-chat `test-inventory.md` and uses the same matrix style as `Discussion_and_announcement_test_matrix_full_with_rules_COMPLETE.md`.

It covers the missing or only-partially-proven tests needed for the implementation-ready libp2p group chat requirements catalog. The matrix intentionally does **not** repeat rows that the current inventory already marks as directly covered unless the remaining requirement is broader than the existing test.

The goal is end-to-end **user trust** for a peer-to-peer group chat:

- users should know who can read, write, invite, remove, ban, acknowledge, or moderate
- topic, peer, relay, sync, and inbox behavior should not leak avoidable metadata
- group state should converge after restarts, partitions, simultaneous actions, and key changes
- messages, media, receipts, notifications, and system events should stay deduplicated and truthful
- cryptographic failures should fail closed, guide repair, and never expose secrets
- unsupported product capabilities should be explicit instead of silently assumed

## Source bundle used

- `test-inventory.md`
- `Discussion_and_announcement_test_matrix_full_with_rules_COMPLETE.md`
- Functional requirements catalog supplied in the prompt, `FR-001` through `FR-380`

## Actors

- **A** = group creator, owner, admin, or authorized moderator
- **B** = active group member on one device
- **C** = invitee, rejoiner, removed member, banned identity, or sibling device
- **D** = prospective member not yet in the group
- **X** = unauthorized peer, stale peer, malicious peer, relay, mailbox, storage peer, or protocol-incompatible peer

## Status legend

- **Partial** = adjacent or lower-level coverage exists in the current inventory, but the full user-visible or protocol-visible requirement is not directly proven.
- **Open** = no dedicated coverage was found in the current inventory for the stated behavior.
- **Unsupported** = the capability appears outside the currently shipped scope; keep it excluded from release gates unless product scope changes.

## Priority guide

- **P0** = release-blocking trust, privacy, safety, authorization, or crypto contract
- **P1** = important before broad rollout or before claiming full group-chat completeness
- **P2** = roadmap or product-dependent capability; add coverage when the capability is intentionally in scope

## Coverage policy used in this matrix

### Coverage legend

- **Required** = should exist before treating the row as production-ready
- **Recommended** = high-value proof but not always required for the first release gate
- **N/A** = do not force this layer for the row

### Rules

**Unit**
Use for deterministic logic: validation, permission checks, state conflict resolution, replay protection, epoch decisions, notification eligibility, error classification, metadata minimization, and API payload shape.

**Integration**
Use for most rows where trust depends on local persistence, application services, bridge payloads, and UI-visible state.

**Smoke**
Keep small and release-blocking. Smoke should prove the core user journey and the highest-risk failure modes, not every branch.

**Fake Network**
Use when the risk is peer discovery, PubSub duplication, relay fallback, sync gaps, partitions, store-and-forward behavior, or malicious/stale peer traffic.

**3-Party E2E**
Use for flows users perceive across A/B/C devices: create, invite, join, removal, key rotation, media, notifications, history sync, and multi-device convergence.

## Current repo execution note (2026-04-29)

The current inventory is strong on discussion and announcement happy paths, direct send/receive, pending invites, admin-only announcement writes, role promotion/demotion, removals, key rotation basics, replay dedupe, inbox drain, retry/resume, media scaffolding, Go-side PubSub validation, and several decryption-failure paths.

The missing coverage is concentrated in broader libp2p requirements, advanced group governance, invite revocation/link flows, full role and permission matrices, bans, device-specific admission, message ordering, edit/delete/tombstone rules, media safety and chunk-resume integrity, MLS-like epoch handling, direct sync protocols, receipts, privacy controls, abuse limits, audit/diagnostics, APIs, and real multi-device or real-network evidence.

## Matrix interpretation

- Rows marked **Partial** should usually be closed by extending an existing test file already named in `test-inventory.md`.
- Rows marked **Open** usually need a new use-case, integration, bridge, Go, fake-network, or E2E suite.
- Rows marked **Unsupported** should remain non-gating until the feature lands, but they are included so scope changes have ready acceptance criteria.
- FR references in the Notes column indicate the requirement group covered by the missing test row.

## Notes for implementation

- Prefer extending existing suites where the inventory already has adjacent coverage, especially `group_message_listener_test.dart`, `send_group_message_use_case_test.dart`, `handle_incoming_group_message_use_case_test.dart`, `accept_pending_group_invite_use_case_test.dart`, `group_key_update_listener_test.dart`, `group_resume_recovery_test.dart`, Go PubSub validator suites, and bridge tests.
- Use fake-network tests for deterministic partitions, duplicate deliveries, stale peers, and relay-only behavior before adding slower real-device E2E.
- Treat any test that inspects payloads, diagnostics, debug export, or push payloads as a privacy test: assert that secrets, plaintext, and sensitive peer addresses are absent, not just that the operation succeeds.
- For **Unsupported** rows, write guard tests only if the API or UI surface exists. Otherwise, keep the row as a product-scope reminder rather than a failing release gate.
