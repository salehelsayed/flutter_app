
# MessageContextOverlay Test Matrix with Coverage Rules

## Scope

This file covers the **MessageContextOverlay** feature triggered from long-pressing a supported message surface and exposing the five overlay options:

- **Reactions**
- **Copy**
- **Reply**
- **Edit**
- **Delete**

The matrix follows the same planning style as the provided introduction matrix, but is adapted for a direct-chat overlay instead of a 3-party introduction flow.

The goal is not only option-by-option completion. The goal is **state convergence and UI correctness**:

- the overlay must show the right actions for the current message state,
- local-only actions must stay local,
- networked actions must converge across sender and receiver under retries, duplicates, reorder, offline delivery, restart, and replay,
- and destructive actions must clean up message artifacts safely without leaving stale UI or orphaned files.

This matrix covers the direct conversation `LetterCard` host and the shared direct-thread feed host where the same overlay primitives are reused.

## Actors

- **A** = local user opening the overlay and triggering an action
- **B** = remote peer in the direct conversation / direct thread
- **X** = unauthorized, stale, blocked, replayed, or mismatched sender/event
- **OS** = local platform service used by local-only flows (for example the system clipboard)

## Priority guide

- **P0** = release-blocking; core permission gates, action correctness, convergence, authorization, cleanup, and no-data-loss guarantees
- **P1** = important; should be covered before broad rollout
- **P2** = optional or polish-heavy; cover before scale-out or when the behavior is highly user-visible

## Minimum MessageContextOverlay rules and invariants

These are the baseline feature rules this matrix assumes:

- The overlay opens only from supported long-press surfaces; deleted/tombstoned rows must not expose mutating actions.
- The context-menu action order is stable: **Reply first**, optional **Edit**, optional **Copy**, optional **Delete last**.
- **Copy** is shown only when `!message.isDeleted && message.text.trim().isNotEmpty`; it is local-only and must not touch repos, bridge, or P2P.
- **Edit** is shown only when all eight UI gate conditions pass: edit enabled, callback wired, not deleted, own identity known, outgoing, owned by local peer, non-empty text, and the message is the latest non-deleted outgoing row.
- **Delete** is shown only when `onDeleteMessage` is wired, the row is not already deleted, and `transport != 'system'`.
- **Delete for everyone** is available only for an outgoing, owned, delivered, non-deleted message when local identity is known.
- **Reply** stores `quotedMessageId`; quote previews and quote bars must resolve message text, media preview text, or a clear unavailable state.
- **Reactions** apply only to non-deleted messages; one reaction per `(message_id, sender_peer_id)`; tapping the same emoji toggles off, and a different emoji replaces the old one.
- **Edit** reuses the original message ID and preserved timestamp / createdAt, adds `editedAt`, and deleted messages never show `(edited)`.
- **Delete for me** hard-deletes the local row and local artifacts only; the peer remains unaffected.
- **Delete for everyone** persists a tombstone, cleans artifacts, transmits a delete payload, and the receiver must authorize the deletion against the original author before applying it.
- Sender-side outgoing delete tombstones stay visible while `sending`, `sent`, or `failed`, and become locally hidden only once the delete is effectively delivered.
- Startup / reconnect reconciliation must apply reply, edit, reaction, and delete state before first meaningful render, not only after the user manually opens the conversation.
- **Critical invariant:** if A deletes a message for everyone before B opens the app or conversation, B must not be able to read the original plaintext on first open.
- Shared conversation and feed direct-thread hosts must behave equivalently wherever the same overlay action is reused.

## Current repo execution note (2026-04-09)

Use this note when turning the matrix into repo-local work:

- This matrix is **coverage-targeting** from the provided C4 documents and the reported delete bug; it does **not** claim these tests already exist.
- The highest-risk reliability rows are the pre-open **Delete for everyone** bug, same-ID **Edit** ordering and authorization, optimistic **Reaction** drift when transport and inbox both fail, and sender-side tombstone visibility / cleanup after **Delete**.
- Rows that are mostly UI-facing should still get **widget coverage** in practice, even though widget tests are not a dedicated column in this matrix.
- Do not let generic `sendChatMessage()` happy-path coverage replace overlay-specific proof for `quotedMessageId`, `editedAt`, delete tombstones, reaction toggles, or overlay dismissal sequencing.

## Coverage policy used in this matrix

### Coverage legend

- **Required** = should exist for this journey before you treat the feature as production-ready
- **Recommended** = high-value coverage, but not mandatory for every release gate
- **N/A** = do not force this layer for this journey

### Rules

**Unit**
Use for logic-heavy pieces:
- action gates and action ordering
- latest-sent scanning for Edit
- toggle / replace semantics for Reactions
- quote-preview resolution
- same-ID edit / delete authorization
- latest-wins ordering / replay protection
- tombstone visibility normalization
- owned-path cleanup guards
- listener lifecycle and duplicate-subscription protection

**Integration**
Use for most journeys:
- overlay -> screen -> wired -> use case -> repository -> listener -> UI round trips
- clipboard write + snackbar behavior
- quote / edit / delete state transitions in the live conversation
- local persistence + receive-side mutation
- cleanup of reactions, attachments, and files

**Smoke**
Keep this small and release-blocking:
- overlay opens and dismisses cleanly
- add / toggle-off reaction
- copy happy path
- reply send with quote rendering on both sides
- edit happy path
- delete for me
- delete for everyone happy path
- pre-open delete regression

**Fake Network**
Use where transport behavior is the main risk:
- retries
- duplicates
- out-of-order edit/delete/reaction delivery
- offline recipient
- inbox/store-and-forward
- cold start / restart reconciliation
- partial delivery
- send=false + inbox=false drift
- late stale traffic after terminal delete

**2-Party E2E (2 simulators)**
Use for user-visible A/B flows:
- A mutates a message and B sees the correct final state
- delete before open / deep-link / cold start
- quoted replies render correctly on both sides
- edits converge on both devices
- reactions converge on both devices
- sender and receiver rebuild the same state after restart or reconnect

### Matrix interpretation

- **Integration** is Required for most rows because overlay reliability emerges across UI, state management, persistence, transport, and receive-side processing.
- UI-dominant rows should still have widget tests even though widget is not a dedicated column.
- Transport-sensitive rows need a deterministic fake network; do not replace them with incidental sleeps or broad smoke tests.

## Missing journeys added in this revision

Compared with a simple happy-path overlay plan, this revision explicitly adds the journeys most likely to explain “works in the demo, fails in production”:

- delete-for-everyone **before the recipient opens the app**
- overlay dismiss -> follow-up sheet sequencing for Delete
- same-ID Edit authorization and stale-edit ordering
- reaction optimistic-state drift when live send and inbox both fail
- delete-before-original and edit-before-original ordering
- reply preview when the quoted source is media-only, deleted, or missing
- cleanup proof for reactions, attachments, and owned file paths
- host parity between ConversationScreen and shared feed direct-thread surfaces
- listener lifecycle / duplicate-subscription protection

## Notes for implementation

- For every mutating journey, assert **four layers together** when applicable: visible UI, local DB/repository state, outbound envelope contents, and receiver-side final state.
- For every fake-network row, make ordering explicit: direct-vs-inbox, live send false vs inbox false, delete-before-original, edit-before-original, duplicate delivery, restart timing, and inbox drain timing should all be deterministic controls.
- For delete rows, verify **cleanup** as a first-class outcome: reactions table, attachment records, local files, quote resolution, active edit state, and active quote state.
- For copy rows, use a fake clipboard and assert the absence of **all** non-local collaborators: no repo, no bridge, no P2P.
- For security rows, craft envelopes with mismatched `message.from`, top-level `senderPeerId`, decrypted payload sender, and target-message author to avoid accidental trust-by-shape.
- For host-parity rows, reuse the same fixtures across `ConversationScreen` and the shared feed direct-thread host so regressions cannot hide behind duplicated test data.
- For the reported delete bug, add at least one **cold-start** and one **notification/deep-link** variant, and assert the first readable frame inside the app is already the tombstone state.
